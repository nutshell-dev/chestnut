# Clawforum 重构原则

## Philosophy

clawforum 的世界观与存在理由,约束一切设计决策的最顶层前提。

* Clawforum 本质是上下文工程
* 分多个智能体、分子任务,目的是更高效地利用上下文窗口,而不是模拟人类组织架构的分工;各个 claw 的特长能力可以轻易复制过来
* Agent 即目录,对话即状态

## Design Principles

运行期行为的原则,约束 clawforum 作为一个 agent 运行时"怎么跑"。

**信息完整性**

* 运行中的所有信息不丢失,状态可观察,中断可恢复,事后可审计
  * 用户可以观察运行过程中的所有状态
  * 运行中断即从最后完整步恢复并继续
  * 运行中产生的所有信息全量记录以供审计

**智能体与系统、智能体之间**

* 智能体是决策主体,执行走受约束路径;创建子代理后立即继续决策,不阻塞等待结果
* 系统只负责给智能体提供决策所需信息和基础设施,系统内部走可信路径
* 多个智能体的信息不应当隔绝,允许互相访问,并且设置特殊智能体 Motion 主动整合多个智能体的持久化记忆充分提取信息

## Module Logic Principles

架构的核心是控制系统复杂度。耦合是系统复杂度扩散的路径。通过管理依赖来控制耦合。

模块逻辑原则要足够泛化,能覆盖未预见的情况,而不是对已知情况的枚举描述。

* 一个模块封装一组独立可变的职责,每种职责只归一个模块
* 模块为自己的业务语义负责,需要业务语义的动作由该模块发起
* 每种资源只归属唯一模块,其他模块通过该模块的对外入口间接访问
* 持久化一切信息到磁盘(运行时句柄从磁盘信息重建)
* 模块的依赖单向,禁止双向/循环,底层模块不预设上层模块语义
* 模块的依赖结构稳定:运行时模块间依赖关系不可变
* 模块的耦合界面稳定：对外表面不随内部实现或外部模块增减而变化
* 模块的耦合界面最小：对外表面只暴露必要的最小集合，跨边界只传必要信息
* 不可消除的耦合应显式表达
* 模块设计中出现依赖问题就要停止当前任务先讨论模块重构

## Coding Principles

代码的目标是正确实现上层设计。编码原则服务于这个目标,降低写出不正确实现的风险。

编码原则要足够泛化,能覆盖未预见的情况,而不是对已知情况的枚举描述。

**结构**

* 减少代码间的依赖：改一处不应连带改多处
* 每一层抽象只隐藏调用方不需要关心的细节

**数据**

* 让状态的变化可预测：可变状态应有唯一且明确的管理者

**错误**

* 预期失败由调用方显式处理,不可预期失败暴露而非吞没

**命名**

* 名字准确反映意图,同一概念用同一名字

**测试**

* 测试验证行为契约,让代码经历从未走过的路径

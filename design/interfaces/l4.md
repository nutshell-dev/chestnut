## TaskSystem [capability, DI]

**生产方**：`l4_task_system`

**消费方**：

- `l5_runtime`（DI，Daemon 触发 initialize 加 startDispatch 加 shutdown）
- `l5_cron`（DI，cron 任务派 TaskSystem 任务）
- `l4_contract_system`（DI，LLM 验收派 verifier 子代理）
- `l4_evolution_system`（DI，派 retro 子代理）
- `l4_memory_system`（DI，dream 触发派 sub-agent）
- `l2_tools`（DI，异步工具 handler 内派任务，如 spawn handler）
- `l6_assembly`（DI，装配期注册 task type executors）

**接口签名**（实然 = class with ctor injection / 双 task type schedule entry / 不立 generic schedule abstract）：

```ts
export class TaskSystem {
  constructor(clawDir: string, fs: FileSystem, options: TaskSystemOptions);

  // 生命周期
  initialize(): Promise<void>;
  startDispatch(): void;
  shutdown(timeoutMs?: number): Promise<void>;       // default 30000

  // 调度（双 entry / 不立 generic schedule(taskType, taskData) 抽象）
  scheduleSubAgent(taskData: Omit<SubAgentTask, 'id' | 'createdAt'>): Promise<string>;  // 返 taskId
  scheduleTool(/* ... 详 src/core/task/system.ts:225 */): Promise<string>;
  writePendingSubAgentTask(/* ... */): Promise<void>;

  // 取消 / 状态
  cancel(taskId: string): Promise<void>;
  listRunning(): string[];
  listPending(): string[];

  // Result handler 注册（pipeline pattern / 4 参 4 args / 返 string 而非 void / 应然原写 TaskResult 应然幻象）
  addTaskResultHandler(
    handler: (taskId: string, callerType: CallerType | undefined, result: string, isError: boolean) => Promise<string>,
  ): () => void;

  // 装配期注入流式输出 sink
  setParentStreamLog(sink: StreamLog): void;
}

export interface TaskSystemOptions {
  maxConcurrent?: number;                           // default DEFAULT_MAX_CONCURRENT_TASKS
  auditWriter: AuditLog;
```

注：`AuditLog` / `StreamLog` 来自 [L2](../l2b.md)，`LLMOrchestrator` 来自 [L2 LLMOrchestrator](../l2b.md#llmorchestrator-capability-di)，`OutboxWriter` 来自 [L2 Messaging](../l2c.md#messaging-capability-di)，`ContractSystem` 来自 [L4 ContractSystem](#contractsystem-capability-di)。

**使用语义**：

- 调度失败原样抛错（应然 silent on `TaskSystemError` 类 / 实然不存在 / 应然幻象删）
- 任务执行失败由 executor 抛错 / 本模块捕获转 failed 状态文件
- 崩溃恢复：重启时 running/ 文件回搬 pending/，result.txt.sent marker 标记过即转 done
- 任务文件 schema 由本模块定义两类 (`SubAgentTask` / `ToolTask`) discriminated union 而非应然 generic `TaskData = Record<string, unknown>`
- 同步工具调用不走本模块（走 L2 Tools router 直接派发）
- PENDING_QUEUE_MAX 加并发上限 operational tuning（装配期注入）
- 任务结果回传走 `addTaskResultHandler` (pipeline pattern / 多 handler 顺序调用 / 后者接前者返回 string)
- handler signature 实然 4 参 (taskId, callerType, result: string, isError) → 返 string / 应然原写 2 参 + TaskResult 类型已删

**应然 silent on `queueLength()` method**（实然不存在 / 应然幻象删）/ **应然 silent on `TaskResult` + `TaskSystemError` + generic `TaskData` type**（实然 0 实施 / 应然幻象删 / SubAgentTask + ToolTask 两 specific 类型替代）。

**归本模块**：clawforum 异步任务调度的唯一入口。业务模块要派异步任务必经本模块。

**不归本模块**：

- 单步 LLM 调用，归 L3 StepExecutor（透过 sub-agent 内部链）
- agent 执行循环，归 L3 AgentExecutor（透过 sub-agent 内部链）
- sub-agent 单实例生命周期管理（启动、超时、abort 响应、parentContext inherit），归 L3 SubAgent
- 系统级长期循环（motion 加 claw 主代理事件循环），归 L5 Runtime
- 定时调度，归 L5 Cron（cron 任务可派 TaskSystem 任务，但调度本身归 Cron）
- 同步工具调用调度，归 L2 Tools router
- 业务级任务语义（具体 task type 执行算法），归各执行器业务

**不可消除耦合理由**：M#1 单一职责（异步任务调度跟单步执行跟循环算法跟单实例生命周期都是独立可变职责）加 M#2 业务语义归属（异步调度加状态机加崩溃恢复加结果回传语义归本模块）derive 自 Design Principle「磁盘即权威」加「中断可恢复」— 异步任务必经本模块目录队列让重启可恢复。

## ContractSystem [capability, DI]

**生产方**：`l4_contract_system`

**消费方**：

- `l5_runtime`（DI，启动期初始化加关闭期清理）
- `l4_evolution_system`（DI，订阅 contract_completed 事件）
- `l2_done_tool`（DI，agent 调 done 工具时 mark contract done）
- `l6_cli`（DI，CLI 命令操作契约 create / pause / resume / cancel / log）
- `l6_assembly`（DI，装配期注册到 Tools 加 inject onContractCompleted callback）

**接口签名**（实然 = class with ctor injection / 多 subtask 概念 / progress 模型 / 不立 simple Contract spec/lifecycle 抽象）：

注：`ContractVerifierScheduler` 来自 `src/core/contract/verifier-scheduler.ts`（port pattern / 装配方注入 verifier subagent 调度实现 / 详 §port pattern feedback 反向）/ `RetroScheduler` 来自 [L4 EvolutionSystem](#evolutionsystem-capability-di）。详 [modules/l4_contract_system.md](../modules/l4_contract_system.md) §A.naming-1。

**使用语义**：

- 实然契约模型 = ContractYaml input + 多 subtask + progress + acceptance / 应然原 simple `ContractSpec` + retryLimit/escalationThreshold 已替换为 multi-subtask 模型
- pause(contractId, **checkpointNote**) / cancel(contractId, **reason**) 多必填参（应然原 silent / 实然要求）
- `loadActive()` / `loadPaused()` 实然方法（应然原 silent / 用于状态查询）
- `getProgress(contractId)` 返 ProgressData / `completeSubtask(params)` 返 AcceptanceResult / 多 subtask 模型核心
- `isComplete(contractId): boolean` 终态判定
- `setOnNotify(cb)` 装配期 wire 通知回调（应然原 silent / 实然 callback inject 模式）
- `onContractCompleted(cb): () => void` event 订阅 + 返 unsubscribe ✓ aligned
- LLM 验收透过 `verifierScheduler` (port pattern) 派 verifier 子代理 / retro 透过 `retroScheduler` (port pattern) 派 retro 子代理
- done 工具实现归本模块 own (`builtins/done.ts`)

**应然 silent on**：
- `done(contractId, result: ContractResult): Promise<void>` 应然幻象（实然由 doneTool 完成 / 不在 ContractSystem class method）
- `list(filter): Contract[]` + `log(contractId): ContractLog` 应然幻象（实然 0 实施 / 多 subtask 模型用 loadActive/loadPaused 替代）
- `ContractSpec` / `AcceptanceCriteria` / `ContractResult` / `ContractFilter` / `ContractLog` / `ContractCompletedHandler` 应然 type（实然完全不同 schema / ContractYaml + ProgressData + AcceptanceResult 替代）
- `ContractLockError` class（实然不存在 / 文件锁失败原样抛 / 应然幻象删）

**归本模块**：clawforum 契约生命周期管理的唯一入口。业务模块要操作契约必经本模块。

**不归本模块**：

- 单步 LLM 调用加 agent 循环（LLM 验收透过 TaskSystem 派 verifier 子代理）
- 跨 agent 通信（escalation 通知加验收结果回传透过 Messaging）
- 契约业务策略（重试次数、escalation 阈值、验收脚本内容），装配期配置
- agent 决策替代（escalation 后 agent 决策怎么应对）
- 契约模板提炼（归 L4 EvolutionSystem 订阅 contract_completed 事件）
- done 工具实现（归各业务模块，本模块提供 done() 接口让工具实现层调）

**不可消除耦合理由**：M#1 单一职责（契约生命周期管理跟 LLM 验收跟跨 agent 通信跟契约模板提炼都是独立可变职责）加 M#2 业务语义归属（契约创建加状态追踪加验收判定加重试加 escalation 加归档语义归本模块）derive 自 Design Principle「智能体是决策主体」加 D11「motion 单向访问」（motion 经契约传递用户意图给 claw 是 D11 + Philosophy P3 多智能体的应用）。

- memory_search 工具 schema + execute 实现 own 在 `core/memory/tools/memory_search.ts`

- `motionContext?: MotionReviewContext` 可选参 motion 审查模式扩展（应然原 silent / 实然为 motion review use case 预留）
- `RetroScheduler` 装配期注入到 ContractSystem（用于派 retro subagent）
- `DISPATCH_SKILLS_SUBDIR` + `DISPATCH_SKILLS_PATH` const 归 EvolutionSystem own（M#3 资源唯一归属）

**归本模块**：clawforum 能力进化触发加协调的唯一入口。业务模块要订阅 contract 完成派 retro 必经本模块。

**不归本模块**：

- 单步 LLM 调用加 agent 循环（retro 业务透过 TaskSystem 派 retro 子代理）
- skill 内容存储（skill 资源 own 在 L2 SkillSystem）
- skill 写入接口（skill 写入由 agent 用 write 工具完成，本模块不暴露 writeSkill 接口）
- 契约生命周期管理（归 L4 ContractSystem，本模块仅订阅 contract_completed 事件）
- 模板内容生成（retro 子代理跑出来的 skill schema 由 retro subagent 决定）
- 跨 claw 实际装配（dispatch 工具透过 SkillSystem 加载 system skill，本模块不参与 dispatch）

**不可消除耦合理由**：M#1 单一职责（能力进化触发跟 retro 子代理执行跟 skill 资源管理跟跨 claw dispatch 都是独立可变职责）加 M#2 业务语义归属（能力提炼触发加去重加 reload 协调语义归本模块）derive 自 Philosophy「各个 claw 的特长能力可以轻易复制过来」加 Design Principle「智能体是决策主体」。

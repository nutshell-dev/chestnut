## AsyncTaskSystem [capability, DI]

**生产方**：`l4_async_task_system`

**消费方**：

- `l5_runtime`（DI，Daemon 触发 initialize 加 startDispatch 加 shutdown）
- `l5_cron`（DI，cron 任务派 AsyncTaskSystem 任务）
- `l4_contract_system`（DI，LLM 验收派 verifier 子代理）
- `l4_evolution_system`（DI，派 retro 子代理）
- `l4_memory_system`（DI，dream 触发派 sub-agent）
- `l6_daemon`（DI，daemon 进程主入口持有 AsyncTaskSystem 句柄供生命周期管理）
- `l6_assembly`（DI，装配期注册 task type executors / mainDialogStore 后期注入）

> 注：spawn / dispatch / ask_caller 工具归 AsyncTaskSystem own（工具归属 L4 自身 / **非反向依赖 L2 Tools 框架** / 同 phase 360 done + phase 416 memory_search + phase 440 send + phase 442 skill 模板）/ 不在本节列 caller 行（曾误列 `l2_tools` / r65 cross-doc audit 修订）。

**使用语义**：

- 生命周期顺序：`initialize()` 必先调（once-only / 复原 running/ tasks）→ `startDispatch()` 启动调度循环（Runtime startDispatch 前任务不启动 / phase 163 强化）→ caller 调 schedule* 派任务 → `shutdown(timeoutMs)` 等 in-flight 完成或 timeout（默认 30s）
- 调度失败原样抛错（应然 silent on `AsyncTaskSystemError` 类 / 实然不存在 / 应然幻象删）
- 任务执行失败由 executor 抛错 / 本模块捕获转 failed 状态文件
- 崩溃恢复：重启时 running/ 文件回搬 pending/，result.txt.sent marker 标记过即转 done
- 任务文件 schema 由本模块定义两类 (`SubAgentTask` / `ToolTask`) discriminated union 而非应然 generic `TaskData = Record<string, unknown>`
- 同步工具调用不走本模块（走 L2 Tools router 直接派发）
- PENDING_QUEUE_MAX 加并发上限 operational tuning（装配期注入）
- 任务结果后处理走 `postProcessor` 声明式字段（phase438 / registry lookup / 替代 pipeline callback closure）
- PostProcessor 签名：`(result, task, isError, fs, audit) => Promise<string>` / standalone function / 0 closure / 重启可恢复
- `cancel(taskId)` 取消单任务 + audit / `listRunning()` + `listPending()` 同步查询当前队列状态（`listPending` 仅返内存 pendingQueue / 文件未被 watcher 拾起前不可见 / 完整 pending 状态读 `tasks/queues/pending/` 目录）
- `addPostProcessor(name, handler)` Assembly 装配期注册 / runtime 不动注册表 / 同 phase 438 declarative pattern
- `setMainDialogStore(store)` Assembly 后期注入（sessionManager 后构造 / phase470）
- `setParentStreamLog(sink)` 装配期注入流式输出 sink（可选业务 / 不影响主调度路径）
- `writePendingSubAgentTask(motionAudit, taskInfo)` motion-only 调用入口（caller 必传 motion 自身 audit / 写 pending 文件至 motion baseDir）

**应然 silent on**：
- `queueLength()` method（实然 0）
- `TaskResult` + `AsyncTaskSystemError` + generic `TaskData` type（实然 0 / `SubAgentTask` + `ToolTask` 两 specific 类型替代）
- `scheduleTool()` callback API（phase432 fs-driven 化后删 / `writePendingToolTaskFile` module-level helper 替代）
- `addTaskResultHandler()` method（实然 0 / phantom）

**归本模块**：clawforum 异步任务调度的唯一入口。业务模块要派异步任务必经本模块。

**不归本模块**：

- 单步 LLM 调用，归 L3 StepExecutor（透过 sub-agent 内部链）
- agent 执行循环，归 L3 AgentExecutor（透过 sub-agent 内部链）
- sub-agent 单实例生命周期管理（启动、超时、abort 响应、parentContext inherit），归 L3 SubAgent
- 系统级长期循环（motion 加 claw 主代理事件循环），归 L5 Runtime
- 定时调度，归 L5 Cron（cron 任务可派 AsyncTaskSystem 任务，但调度本身归 Cron）
- 同步工具调用调度，归 L2 Tools router
- 业务级任务语义（具体 task type 执行算法），归各执行器业务

**不可消除耦合理由**：M#1 单一职责（异步任务调度跟单步执行跟循环算法跟单实例生命周期都是独立可变职责）加 M#2 业务语义归属（异步调度加状态机加崩溃恢复加结果回传语义归本模块）derive 自 Design Principle「磁盘即权威」加「中断可恢复」— 异步任务必经本模块目录队列让重启可恢复。

- 实然契约模型 = ContractYaml input + 多 subtask + progress + acceptance / 应然原 simple `ContractSpec` + retryLimit/escalationThreshold 已替换为 multi-subtask 模型
- pause(contractId, **checkpointNote**) / cancel(contractId, **reason**) 多必填参（应然原 silent / 实然要求）

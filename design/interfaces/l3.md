## StepExecutor [capability, DI]

**生产方**：`l3_step_executor`

**消费方**：

- `l3_agent_executor`（DI，每轮循环调单步）

**接口签名**（实然 = standalone 自由函数 / 不立 StepExecutor interface / 不持状态）：

```ts
export function executeStep(input: StepInput): Promise<StepResult>;

export interface StepInput {
  messages: Message[];                        // caller own buffer / 本函数 in-place 追加（M#7 anchor 不可改不可变）
  systemPrompt: string;
  llm: LLMOrchestrator;
  tools: ToolDefinition[];                    // 来自 L2 ToolProtocol
  executor: IToolExecutor;                    // 来自 L2 Tools
  registry?: ToolRegistry;                    // 来自 L2 Tools
  ctx: ExecContext;                           // 来自 L2 ToolProtocol
  maxTokens?: number;                         // 默认 REACT_DEFAULT_MAX_TOKENS
  idleTimeoutMs?: number;                     // 透传 LLMOrchestrator
  callbacks?: StepCallbacks;
}

export type StepResult =
  | { kind: 'final'; stopReason: 'end_turn' | 'max_tokens_text' | 'no_tool' | 'unknown'; finalText: string }
  | { kind: 'continue'; meta: StepMeta }
  | { kind: 'max_tokens_tool_use'; meta: StepMeta };

export interface StepMeta {
  toolCallCount: number;
  parseErrorCount: number;
  allParseErrors: boolean;
  llm: LLMCallInfo;
}

export interface LLMCallInfo {
  model: string;
  inputTokens: number;
  outputTokens: number;
  latencyMs: number;
  error?: string;
}

// 12 个细粒度回调（不立单一 onEvent / discriminated union）
export interface StepCallbacks {
  onBeforeLLMCall?: () => void;
  onLLMResult?: (info: LLMCallInfo) => void;
  onTextDelta?: (delta: string) => void;
  onTextEnd?: () => void;
  onThinkingDelta?: (delta: string) => void;
  onToolCall?: (toolName: string, toolUseId: string) => void | Promise<void>;
  onToolResult?: (toolName: string, toolUseId: string, result: ToolResult) => void;
  onReset?: (provider: string, timeoutMs: number) => void;
  onProviderFailed?: (provider: string, model: string, error: string) => void;
  onEmptyResponse?: (stopReason: string) => void;
  onUnknownStopReason?: (stopReason: string) => void;
  onUnparseableToolUse?: (stopReason: string) => void;
}
```

注：`Message` / `LLMOrchestrator` / `ToolDefinition` 来自 [L2](./l2b.md)，`IToolExecutor` / `ToolRegistry` / `ExecContext` / `ToolResult` 来自 [L2 ToolProtocol + Tools](./l2b.md#toolprotocol-type-only)。abort 直抛三 signal class（来自 `src/types/signals.ts` cross-cutting）：`UserInterrupt`（user abort）/ `IdleTimeoutSignal`（idle timeout）/ `PriorityInboxInterrupt`（step_yield）/ **应然 silent on `StepAbortError` 包装类**（应然幻象 / 实然 0 此类 / 直抛 signal class / 应然 rule 必有现实功能依据反向）。

**使用语义**：

- LLM 调用失败由 L2 LLMOrchestrator 抛 typed error，本函数原样传播
- 工具派发失败由 L2 Tools 抛 typed error，本函数原样传播
- abort 直抛三 signal class：`UserInterrupt` / `IdleTimeoutSignal` / `PriorityInboxInterrupt`（src/types/signals.ts / 0 包装类 / caller catch instanceof 区分）
- 单步执行：调用方提供 messages buffer，本函数 **in-place 追加** (M#7 anchor / 不可改为返回新 messages 不可变模式 / 防 caller 语义破坏)
- StepResult 三态 discriminated union：`final` (turn 结束 / 含 finalText) / `continue` (步进 / 含 meta) / `max_tokens_tool_use` (max_tokens 截在 tool_use 边界 / 不步进 / 不熔断 / 调用方累计后熔断)
- 12 细粒度回调 (vs 单一 onEvent) / caller 按需订阅 / 事件命名空间归 caller

**归本模块**：clawforum agent 单步执行的唯一入口。业务模块要跑单 step（一次 LLM 调用加随之而来的 tool 派发）必经本模块。

**不归本模块**：

- LLM 调用容错（重试加 failover 加协议错误识别），归 L2 LLMOrchestrator
- 工具实现加权限校验加审计，归 L2 Tools
- 循环算法（多步调度加停止判定），归 L3 AgentExecutor
- dialog 持久化，归 L2 DialogStore（调用方装配期决定何时落盘）
- agent 业务身份（motion / claw / sub-agent identity），归调用方
- 事件命名空间加业务事件分类，归调用方

**不可消除耦合理由**：M#1 单一职责（单步执行跟循环算法跟工具实现跟 LLM 容错都是独立可变职责）加 M#2 业务语义归属（单步 LLM 调用加 tool 派发的业务语义归本模块）derive 自 Philosophy「clawforum 本质是上下文工程」— 单步是 messages → 新 messages 的纯函数式执行单位，本模块封装让循环算法跟 sub-agent 编排都基于同一执行原语。

## AgentExecutor [capability, DI]

**生产方**：`l3_agent_executor`

**消费方**：

- `l5_runtime`（DI，motion / claw 主代理事件循环）
- `l3_sub_agent`（DI，sub-agent 内部跑一次完整循环）

**接口签名**（实然 = standalone 自由函数 / 不立 AgentExecutor interface）：

```ts
export function runAgent(input: AgentInput): Promise<AgentResult>;

export interface AgentInput {
  messages: Message[];                            // caller own buffer / 在 stepCallback 内落盘
  systemPrompt: string;
  llm: LLMOrchestrator;
  tools: ToolDefinition[];
  executor: IToolExecutor;
  registry?: ToolRegistry;
  ctx: ExecContext;
  maxSteps?: number;                              // 默认 20
  maxConsecutiveParseErrors?: number;             // 默认 MAX_CONSECUTIVE_PARSE_ERRORS=3
  maxConsecutiveMaxTokensToolUse?: number;        // 默认 MAX_CONSECUTIVE_MAX_TOKENS_TOOL_USE=3
  maxTokens?: number;                             // 透传 executeStep
  idleTimeoutMs?: number;                         // 透传 LLMOrchestrator
  stepCallbacks?: StepCallbacks;                  // 来自 StepExecutor / 透传
  onAfterStep?: (meta: StepMeta) => void | Promise<void>;
}

export interface AgentResult {
  finalText: string;                              // 注：返 finalText 而非 messages（caller own messages buffer / 在 stepCallbacks 内自取）
  stepsUsed: number;
  stopReason: 'end_turn' | 'max_tokens_text' | 'no_tool' | 'unknown';
}

// 错误类（cross-cutting）
export class MaxStepsExceededError extends Error {
  readonly maxSteps: number;
}

export class ConsecutiveParseErrorsExceededError extends Error {
  readonly count: number;
  readonly toolNames: string;                     // 最近 assistant message tool_use blocks 拼接
}

export class ConsecutiveMaxTokensToolUseError extends Error {
  readonly count: number;
}

// 公共 API: runReact（agent-executor module 公共 production API / 11 平铺回调形态 / wrap runAgent internal core / Runtime + SubAgent caller-side 装配入口 / per phase 522 ν / single public entry / runAgent 改 module internal core 不进 barrel）
export function runReact(options: ReactOptions): Promise<ReactResult>;

export interface ReactOptions {
  messages: Message[];
  systemPrompt: string;
  llm: LLMOrchestrator;
  executor: IToolExecutor;
  ctx: ExecContext;
  maxSteps?: number;                                         // 默认 20
  maxConsecutiveParseErrors?: number;
  maxConsecutiveMaxTokensToolUse?: number;
  idleTimeoutMs?: number;
  tools?: ToolDefinition[];
  registry?: ToolRegistry;
  // callback set（22 项 / StepCallbacks superset / 多 onToolResult 含 step+maxSteps 旧 signature + onStepComplete 旧 hook）
  onToolCall?: (toolName: string, toolUseId: string) => void | Promise<void>;
  onBeforeLLMCall?: () => void;
  onToolResult?: (toolName: string, toolUseId: string, result: ToolResult, step: number, maxSteps: number) => void;
  onStepComplete?: () => Promise<void>;
  onTextDelta?: (delta: string) => void;
  onTextEnd?: () => void;
  onThinkingDelta?: (delta: string) => void;
  onReset?: (provider: string, timeoutMs: number) => void;
  onProviderFailed?: (provider: string, model: string, error: string) => void;
  onLLMResult?: (info: LLMCallInfo) => void;
  onEmptyResponse?: (stopReason: string) => void;
  onUnknownStopReason?: (stopReason: string) => void;
  onUnparseableToolUse?: (stopReason: string) => void;
}

export interface ReactResult {
  finalText: string;
  stepsUsed: number;
  stopReason: 'end_turn' | 'max_steps' | 'no_tool' | 'max_tokens';
}
```

注：`StepCallbacks` / `StepMeta` 来自 [StepExecutor](#stepexecutor-capability-di)，错误类是 cross-cutting 类型。

**使用语义**：

- 单步抛错原样传播（含 abort）
- 超 maxSteps 抛 `MaxStepsExceededError` (注：应然原写 `AgentExecutorMaxStepsError` / 实然命名 `MaxStepsExceededError` / 应然 align 实然名)
- 连续 parse errors 超阈值抛 `ConsecutiveParseErrorsExceededError` (含 toolNames 字符串供错误诊断)
- 连续 max_tokens_tool_use 超阈值抛 `ConsecutiveMaxTokensToolUseError`
- `onAfterStep(meta)` callback 是「每步算法完成 + 步进 + 熔断后」hook / 调用方在内决定副作用（如 dialog 落盘 / stream emit）
- `stepCallbacks` 12 细粒度回调透传 StepExecutor（不持业务语义）
- 本函数不直接依赖 DialogStore（每步落盘归调用方）
- maxSteps 阈值是守卫 / 调用方注入业务化数值（应然原写 RunParams.maxSteps required / 实然 optional 默认 20）
- 熔断阈值 (`maxConsecutiveParseErrors` / `maxConsecutiveMaxTokensToolUse`) caller 注入 
- `ctx.signal` 由调用方提供（Daemon 装配期注入）/ 本函数在每步前后检查
- `ctx.incrementStep()` 步进通过 ExecContext 自带方法 / 本函数调用 / `ctx.stepNumber` 在 ExecContext 上自累

**归本模块**：clawforum agent 循环算法的唯一入口。业务模块要跑完整 agent 执行循环必经本模块。

**不归本模块**：

- 单步 LLM 调用加工具派发，归 L3 StepExecutor
- dialog 持久化（每步落盘加恢复），调用方在 stepCallback 内调 L2 DialogStore
- 业务事件 emit（stream events 加 audit），调用方在 stepCallback 内决定
- agent 业务身份（motion / claw / sub-agent identity），归调用方
- 业务停止信号语义（何时发 abort），调用方决定（如 Daemon 监听 SIGTERM 触发 abort）
- 跨 LLM 调用容错，归 L2 LLMOrchestrator
- maxSteps 阈值业务化（具体数值来源），归调用方装配期注入

**不可消除耦合理由**：M#1 单一职责（循环算法跟单步执行跟 dialog 持久化跟 agent 业务身份都是独立可变职责）加 M#2 业务语义归属（循环算法跟停止判定语义归本模块）derive 自 Design Principle「智能体是决策主体」加「中断可恢复」— 循环算法本身不持业务语义，调用方通过 stepCallback hook 注入副作用，让本模块跟 dialog 持久化加 agent 身份解耦。

## SubAgent [capability, DI]

**生产方**：`l3_sub_agent`

**消费方**：

- `l4_task_system`（DI，子代理任务执行器）
- `l4_contract_system`（DI，验收路径 / phase 427 后 ContractSystem 内联 `_runVerifierSubagent` 直创 SubAgent / 替代已删 verifier-scheduler.ts）

**接口签名**（实然 = class with ctor injection / `run()` no args returns string）：

```ts
export class SubAgent {
  constructor(options: SubAgentOptions);
  run(): Promise<string>;                          // 返 final text / 不返 SubAgentResult rich type
}

export interface SubAgentOptions {
  agentId: string;
  resultDir: string;                               // phase443: caller 注入完整 path / L3 SubAgent 0 知 tasks/ 字符串约定 / 应然 phase 507+：async caller (AsyncTaskSystem) → `tasks/queues/results/<task-id>/` / sync caller (ContractSystem verifier-job) → `tasks/sync/spawn/<agentId>/` / phase 510-511 实施
  messageStore: DialogStore;                       // phase453: caller 装配期注入 ephemeral DialogStore（filename='messages.json' / 0 clawId / 0 archive 触发）/ messages 经 L2 DialogStore 持久化
  prompt: string;
  clawDir: string;
  llm: LLMOrchestrator;
  registry: ToolRegistryImpl;
  fs: FileSystem;
  taskStreamWriter: StreamLog;                     // 必传（不是 NoopStreamWriter）
  auditWriter: AuditLog;                           // 必传 / 落 resultDir/audit.tsv（应然 phase 507+ tasks/queues/results/ 或 tasks/sync/spawn/ 各按 caller 模式）
  // 可选
  maxSteps?: number;                                // DEFAULT_MAX_STEPS
  timeoutMs?: number;                               // SUBAGENT_TIMEOUT_MS (5 min default)
  signal?: AbortSignal;                             // 外部 abort 链接
  toolsForLLM?: ToolDefinition[];                   // pre-filtered 工具列表 / 覆盖 registry.getAll()
  idleTimeoutMs?: number;
  onIdleTimeout?: () => void;
  maxConsecutiveParseErrors?: number;
  maxConsecutiveMaxTokensToolUse?: number;
  systemPrompt?: string;                             // 覆盖 run() 硬编码默认 system prompt
  callerType?: CallerType;                           // 默认 'subagent'
  // taskSystem 字段 phase 438 后 deprecated（addTaskResultHandler API 删 / postProcessor declarative schema 替代 / 字段保留向后兼容 / 推 r+1 phase 删）
  taskSystem?: AsyncTaskSystem;                           // ⚠ DEPRECATED phase 438 / 0 use / 推 r+1 删
  subagentMaxSteps?: number;                         // 传给嵌套 SubAgent
  messages?: Message[];                              // 实然 copy mode（main messages 复制进 subagent / 应然 marker mode 推 phase 458 改 mainContextSnapshot + ask_caller / 见下应然 NEW 字段）
  originClawId?: string;                             // 创建链路源头 (motion 直创 = 'motion')

  // phase 458 应然 NEW（spawn cluster ask_caller 工具实施前置 / 用户 8 轮 derive 锁定 marker mode）
  mainContextSnapshot?: { clawId: string; toolUseId: string };  // spawn 创建期 capture / 替代 messages copy / 单一权威源 DialogStore
  mainDialogStore?: DialogStore;                                 // read-only ref to main DialogStore / ask_caller 工具用 restorePrefix(marker) / subagent profile 才需要
}

// 工厂（thin proxy 工厂）
export function createSubAgent(opts: SubAgentOptions): SubAgent;

// Noop helpers（测试 / 装配期可选 sink）
export class NoopStreamWriter implements StreamLog {
  write(event: StreamEvent): void;                   // 0-op
}

export class NoopAuditWriter implements AuditLog {
  write(type: string, ...cols: (string | number)[]): void;  // 0-op
}
```

注：`ToolRegistryImpl` 来自 [L2 Tools](./l2c.md#tools-capability-di)，`AsyncTaskSystem` 来自 [L4 AsyncTaskSystem](./l4.md#tasksystem-capability-di)（仅 deprecated 字段 ref），`StreamLog` / `StreamEvent` 来自 [L2 Stream](./l2b.md#stream-capability-di)，`CallerType` 来自 [L2 ToolProtocol](./l2b.md#toolprotocol-type-only)。

**使用语义**：

- 构造期通过 ctor 注入所有依赖（应然不立 SubAgentInput / 配置走 ctor）
- `run()` 无参数 / 返 `Promise<string>` final text（应然原写 `SubAgentResult { finalMessages, finalStopReason, dialogPath }` / 实然返 string only）
- 超时（total + idle 双窗口）走 AbortController + audit `TURN_TIMEOUT` / `IDLE_TIMEOUT` / 不抛 SubAgentTimeoutError 类（应然幻象删）
- LLM 加循环抛错原样传播
- 生命周期 events（`turn_start` / `turn_end` / `turn_error` 等）必经 `taskStreamWriter` + `auditWriter`
- 内部组装 ExecContext + 调 runAgent / runReact 完成 one-shot 循环
- one-shot：跑完即结束（长期循环归 L5 Runtime）
- 崩溃恢复归调用方（L4 AsyncTaskSystem own crash recovery）
- `NoopStreamWriter` / `NoopAuditWriter` 装配期可选注入（测试或 sink 不需要的场景）
- `messageStore` phase 457 后必含 systemPrompt（caller 装配期传 subagent 自己 system prompt 进 ephemeral DialogStore ctor / messageStore.systemPrompt = subagent 系统 prompt / 与 main system 不同）
- `mainContextSnapshot` + `mainDialogStore` phase 458 应然 NEW（subagent profile 用 / ask_caller 工具实施前置 / mainDialogStore 是 read-only ref 指向 main 的 DialogStore instance / restorePrefix(marker) 返 main 在 spawn 时刻完整前缀 messages + systemPrompt）

**应然 silent on `SubAgentInput`/`SubAgentResult`/`SubAgentTimeouts`/`SubAgentIdentity`/`SubAgentTimeoutError` 5 应然幻象 type**（实然 0 这些 class / 全走 ctor + return string + AbortController + audit events / 应然 rule 必有现实功能依据反向）。

**归本模块**：clawforum sub-agent 单实例生命周期管理的唯一入口。业务模块要派一个 sub-agent 跑一次必经本模块。

**不归本模块**：

- 循环算法本身，归 L3 AgentExecutor（本模块内部调用）
- 单步执行，归 L3 StepExecutor（透过 AgentExecutor 间接调用）
- system prompt 构造（直接写加从 skill 加载加用模板等），调用方提供 systemPrompt
- tools 子集装配（哪些 tool 给 sub-agent），调用方提供 tools instance
- 排队加并发调度，归 L4 AsyncTaskSystem
- 崩溃恢复，归 L4 AsyncTaskSystem
- 父子代理调度关系（parentContext 内容、result 投回父 inbox、cascade abort），归 L4 AsyncTaskSystem
- 长期循环（motion / claw 主代理），归 L5 Runtime
- LLM 直调，归 L2 LLMOrchestrator（透过 AgentExecutor 加 StepExecutor 间接）

**不可消除耦合理由**：M#1 单一职责（sub-agent 单实例生命周期跟循环算法跟跨 sub-agent 调度跟主代理长期循环都是独立可变职责）加 M#2 业务语义归属（sub-agent 实例化加生命周期 events 加超时约束加父子上下文继承的业务语义归本模块）derive 自 Philosophy「分多个智能体加分子任务」加 Design Principle「创建子代理后不阻塞，结果异步返回」— sub-agent 业务语义不能让循环算法（AgentExecutor）持有，否则破坏 AgentExecutor「不持业务语义」契约。

## Callback signatures

### onStepEvent — L3 StepExecutor own，调用方实现

执行过程事件透传给调用方。事件命名空间加业务分类归调用方。signature 见 StepExecutor 的 `StepParams.onEvent`。

### onAfterStep / stepCallback — L3 AgentExecutor own，调用方实现

每步算法完成后调 callback，调用方在 callback 内决定每步副作用（如 dialog 落盘、stream emit、audit 写入）。本 hook 是 AgentExecutor「不持业务语义」契约的核心实现机制。signature = `AgentInput.onAfterStep?: (meta: StepMeta) => void | Promise<void>`（命名 align 实然 / modules 历史用「stepCallback」是同一概念）。

**典型实现方**：

- L5 Runtime 实现 stepCallback：调 L2 DialogStore 落盘 messages 加 L2 Stream emit 执行过程事件
- L3 SubAgent 实现 stepCallback：调 L2 DialogStore 落盘 sub-agent 独立 dialog

# Clawforum 模块清单

## 模块分层

| 层 | 模块 |
|---|---|
| L1 原语（OS / 外部系统抽象层） | FileSystem, FileWatcher, ProcessExec, LLMService, Transport |
| L2 基础设施 | **纯通用**（模块代码不知 agent 业务）: ProcessManager, AuditLog, Stream, Snapshot；**agent 语义**（含 agent 业务概念）: SkillSystem, SessionStore, Messaging, Tools, FileTool, ShellTool |
| L3 执行与连接 | StepExecutor, AgentExecutor, SubAgent, Gateway |
| L4 任务与业务 | TaskSystem, ContractSystem |
| L5 外壳与能力 | Runtime, Cron, MemorySystem, ContractRetro |
| L6a 进程入口 | Daemon, Watchdog |
| L6b 交互入口 | CLI |
| L6c 装配 | Assembly |

共 29 个模块。

### L1 定位（OS / external system 抽象层）

**only**：
- 直接 wrap stdlib / OS API / 网络 / 外部服务
- 暴露中性接口（隐藏 OS / 协议 / 实现差异）
- 跨 OS 时唯一需要 OS 分支的层

**never**：
- 不依赖任何其他 clawforum 模块（含 L1 同层 / L2+ 上层）
- 不 own audit（cross-cutting 业务 / caller 自己 audit）
- 不 own 权限 / 安全 / 路径限制（agent 业务 / L2 agent 语义 wrapper own）
- 不 own agent 工具导出（agent 业务 / L2 agent 语义 wrapper own）
- 不 own 装配 / 生命周期编排（L6 装配 own）
- 不预设上层语义（chokidar 不知 stream / fs 不知 session / spawn 不知 daemon）

L1 当前 5 模块：FileSystem / FileWatcher / ProcessExec / LLMService / Transport。

## 系统拓扑

```
        用户
         ↕（经 Transport + Gateway）
       motion                   （特殊 claw：对外数据面 + 对内整合者）
      ↕ CLI / Messaging / 跨 claw 资源读（motion 单向访问权）
  claw₁  claw₂  ...            （执行 agent；无用户直连）

独立进程：Watchdog（观察 motion + claw 健康状态）
```

**装配归属维度**：每模块条目的「装配归属」字段精确指明该模块的可装配性约束。可能值：

| 值 | 含义 |
|---|---|
| `按需` | 任何 daemon 装配方按 use case 决定是否装 / 多少实例（绝大多数模块）|
| `独立进程` | 自成进程，架构强制（如 Watchdog 跨 daemon health monitor / 不能装入被监控对象内）|
| `命令进程` | 短生命周期命令进程（如 CLI / 不装入 daemon 是因为命令式入口 vs long-running service 的不同）|

**原则**：
- 装配决策查此维度 + 拓扑图，不凭直觉
- 模块边界与拓扑不一致 → 按原则 #11「边界和实际依赖对不上停下来讨论」启动重构

## L1 原语

### 1. FileSystem

clawforum 内部代码的所有文件 I/O 唯一入口。原子写。

- **装配归属**：按需（文件 I/O 原语 / 任何使用磁盘的 daemon 都装）
- **资源**：无自有资源 / 提供文件操作能力
- **依赖**：无
- **耦合**：无
- **对外能力**：基本文件读写 / 原子写 / 独占创建 / gitignore 模式常量
- **被谁调用**：几乎所有模块
- **对外接口契约**：[modules/l1_filesystem.md](modules/l1_filesystem.md)

### 2. ProcessExec

进程 OS 能力的统一 wrapper / 跨 OS 抽象层。

- **装配归属**：按需
- **资源**：无
- **依赖**：无
- **耦合**：无
- **对外能力**：exec 短任务 / detached spawn / kill 中性信号 TERM/KILL/INT / 存活检查 / pattern 查找 / 内部按 OS 分支 / 不暴露 shell mode（caller 自负 shell 跨 OS 风险）
- **被谁调用**：所有需要调用外部进程的模块（应然 ProcessManager L2 + ShellTool L2 + ContractSystem L4 验收脚本 等）
- **对外接口契约**：[modules/l1_process_exec.md](modules/l1_process_exec.md)

### 3. LLMService

LLM 调用的统一服务。provider 管理 / 请求组装 / KV cache 标记 / 重试与超时。

- **装配归属**：按需（跑 agent 的 daemon 装）
- **资源**：无
- **依赖**：无
- **耦合**：定义事件发布协议，由装配方提供实现把事件 fan-out 到审计与流式观察
- **定义的协议**：`LLMEventSink`（provider 失败 / 退避 / breaker 状态 / failover / healthcheck / stream_reset 事件发布协议）
- **对外能力**：一次性调用 / 流式调用 / 健康探测 / provider 状态查询 / primary+fallbacks 重试 / circuit breaker / abort / 中途失败 reset
- **唯一消费者**：StepExecutor
- **对外接口契约**：[modules/l1_llm_service.md](modules/l1_llm_service.md)

### 4. Transport

实时双向持久连接原语。管理 connection 生命周期 / 提供消息推送 + 接收能力。具体协议（socket / pipe / WebSocket / 等）是内部实现细节 / 契约层不暴露。

- **装配归属**：按需（任何需要持久连接通信的 daemon 装）
- **资源**：无磁盘资源；维护连接表 + 接收缓冲（派生态 / 重启从 socket 握手重建）
- **依赖**：无
- **耦合**：定义不可预期失败通道协议 / 装配方接收后通常 fan-out 到 AuditLog / Stream
- **定义的协议**：`TransportErrorEvent`（不可预期失败通道协议 / 含回调抛错 + server-level 错误两类）
- **对外能力**：监听启停 / 定向发送 / 广播（best-effort）/ 连接生命周期回调订阅 / 消息接收回调订阅 / 不可预期失败通道订阅
- **被谁调用**：装配方提供的连接消费者（当前实然 Gateway 是唯一上游 / 未来扩展时其他需要持久连接通信的模块可调用）
- **对外接口契约**：[modules/l1_transport.md](modules/l1_transport.md)

### 5. FileWatcher

文件系统变化通知 OS 原语。chokidar wrap（inotify / FSEvents / ReadDirectoryChangesW 跨 OS 抽象）。不懂业务目录语义 / 不读文件内容 / 不 own audit。

- **装配归属**：按需（任何需要订阅文件变化的模块装）
- **资源**：无（订阅集合是运行期状态 / 重启重建）
- **依赖**：无
- **耦合**：无
- **对外能力**：路径变化订阅 / 事件类型映射 / 初始扫描完成信号 / 原始错误 callback（caller 自己 audit）/ 稳定性窗口可选 / 句柄 close idempotent
- **被谁调用**：所有需要订阅文件变化的模块（caller 拿到 raw `onError` callback 后按自己业务上下文 audit / event 命名空间归 caller）
- **对外接口契约**：[modules/l1_file_watcher.md](modules/l1_file_watcher.md)

## L2 基础设施

### 6. SessionStore

messages 数组的持久化读写。服务于"中断可恢复"。

- **装配归属**：按需（任何持有 agent 对话 session 的 daemon 装）
- **资源**：current.json + archive/ + .corrupted（父目录由调用方指定）
- **依赖**：FileSystem, AuditLog
- **耦合**：无
- **对外能力**：当前会话读写 / 归档 / 冷启动 archive 恢复 / 损坏文件隔离 / 悬空 tool_use 修复
- **被谁调用**：AgentExecutor（每次 LLM 调用后落盘 / 启动时恢复）
- **对外接口契约**：[modules/l2_session_store.md](modules/l2_session_store.md)

### 7. Stream

执行过程的实时观察窗口。事件流的 schema + 读写 + 同进程 in-process pub/sub。

- **装配归属**：按需（写侧由产生事件的 daemon 装 / 读侧由消费 stream 的模块装）
- **资源**：stream.jsonl（当前会话事件流）、logs/stream/（归档）
- **依赖**：FileSystem
- **耦合**：无
- **定义的协议**：`StreamCallbacks`（事件发布协议 / 上游持 StreamWriter 的所有事件 publisher 通过此接口发布事件）
- **对外能力**：事件追加写 / 启动期归档 / 归档裁剪 / 历史一次性读 / 同进程 in-process pub/sub callback / 事件 schema 解析（bytes → events[]）/ gitignore 模式常量
- **被谁调用**：所有需要发布执行过程事件的模块（写侧）/ 所有需要订阅或回放执行过程事件的模块（读侧 / 跨进程订阅者自己 own FileWatcher 监听 stream.jsonl 追加 + 用 Stream 解析 bytes）
- **对外接口契约**：[modules/l2_stream.md](modules/l2_stream.md)

### 8. AuditLog

状态迁移审计记录。纯追加写 / generic writer / 不知模块语义 / 不维护 event name 清单。

- **装配归属**：按需
- **资源**：audit.tsv
- **依赖**：FileSystem
- **耦合**：无
- **对外能力**：结构化事件追加 / 时间戳由本模块生成 / TSV 字段转义 / 按大小轮转 / 写失败不阻塞业务 / gitignore 模式常量
- **被谁调用**：所有需要审计的模块
- **对外接口契约**：[modules/l2_audit_log.md](modules/l2_audit_log.md)

### 9. Messaging

inbox/outbox 的目录管理 + 消息投递与拉取。所有跨 agent 消息通信的必经之路。

- **装配归属**：按需
- **资源**：inbox/(pending/done/failed)、outbox/(pending)
- **依赖**：FileSystem, AuditLog（inbox/outbox 编解码 + frontmatter 解析为模块内部 helper / 不暴露独立模块）
- **耦合**：无
- **对外能力**：outbox 写入 / inbox 写入（同步 + 异步）/ inbox 排空 + 优先级排序 / 已处理归档 / 失败归档 / frontmatter meta 轻量读取 / inbox/outbox 编解码（内部 helper）
- **导出工具**：send
- **被谁调用**：Runtime / TaskSystem / ContractSystem / MemorySystem / CLI
- **对外接口契约**：[modules/l2_messaging.md](modules/l2_messaging.md)

### 10. ProcessManager

daemon 进程业务编排 + PID 资源管理。通过 ProcessExec 调所有 OS 进程能力 / 不直接 import OS 进程 API / 不绑 POSIX 信号名。

- **装配归属**：按需（任何需要管理子进程或自注册 PID 的进程装）
- **资源**：所有进程的 PID 文件 / lockfile
- **依赖**：ProcessExec, FileSystem, AuditLog
- **耦合**：无
- **对外能力**：daemon spawn 编排 / PID 文件管理 / 存活监控 / 优雅停止（TERM → wait → KILL）/ 孤儿清理 / 排他锁
- **被谁调用**：Daemon / Watchdog / CLI
- **对外接口契约**：[modules/l2_process_manager.md](modules/l2_process_manager.md)

### 11. Snapshot

目标目录的版本化快照（generic git wrapper）。对指定目录执行 git commit / 记录历史状态 / 支持回滚。模块代码不知 agent 业务 /「agent 目录」是典型 caller use case 而非 Snapshot 业务。

- **装配归属**：按需
- **资源**：目标目录内的 .git
- **依赖**：ProcessExec, FileSystem, AuditLog
- **耦合**：gitignore 模式聚合（条目命名归各源模块 / Snapshot 通过参数接收 / Assembly 装配时聚合）
- **对外能力**：幂等 init 写入 .gitignore + 身份配置 / 单次 commit（无变更跳过）/ 连续失败累计降级告警
- **被谁调用**：Runtime / Daemon / CLI
- **对外接口契约**：[modules/l2_snapshot.md](modules/l2_snapshot.md)

### 12. SkillSystem

技能元信息注册表。扫描 skillsDir 加载技能元信息（frontmatter）/ 渐进式披露：启动仅元信息 / 调 skill 工具时才加载完整 SKILL.md。

- **装配归属**：按需
- **资源**：skillsDir 参数指向的目录（默认 'skills'）/ 磁盘即权威 / 内存元信息表派生态可重建
- **依赖**：FileSystem, AuditLog（frontmatter 解析为项目通用 utility / 不入 modules.md 应然层）
- **耦合**：上下文摘要格式化作为耦合窄化点收敛上下文注入格式
- **对外能力**：技能元信息扫描加载 / 单名查 + 列全 / 渐进式完整内容加载 / 上下文摘要生成
- **导出工具**：skill（dispatch 工具消费上下文摘要 / dispatch 本身归 TaskSystem L4 导出）
- **被谁调用**：Assembly / Runtime / ContextInjector / TaskSystem / SubAgent / skill 工具
- **对外接口契约**：[modules/l2_skill_system.md](modules/l2_skill_system.md)

### 13. Tools

Agent 工具调用机制框架（L2 agent 语义）。提供 Tool 协议 + 注册表 + dispatcher（含超时 / async 路由 / 调用 audit）+ 执行上下文 + caller 权限矩阵。**不持任何具体工具实现**（read/write/exec/send/skill/done/spawn/dispatch 等业务工具由各业务模块 own + 装配期注册）。

- **装配归属**：按需
- **资源**：无（注册表是运行期派生态 / 不落盘）
- **依赖**：AuditLog（dispatcher generic 调用 audit）
- **装配源**：业务模块定义的 Tool 对象作为注册源（装配期由 Assembly 注册）
- **耦合**：无业务能力协议
- **定义的协议**：`Tool`（工具实现形状契约）/ `ToolRegistry`（注册表契约）/ `ToolDispatcher`（dispatcher 契约 / 含超时 + signal + audit）/ `ExecContext`（执行上下文契约 / 含 caller 身份 + 依赖句柄）/ `CallerType` + `TOOL_PROFILES`（caller 权限矩阵契约）
- **对外能力**：工具注册 / 卸载 / 查询 / 按权限过滤 / LLM 格式转换 / 工具调用（同步 + async 路由 / generic audit / timeout / abort）/ 并行优化（readonly 工具批量并发）
- **被谁调用**：StepExecutor / AgentExecutor / SubAgent / TaskSystem / ContractSystem / Runtime / Assembly
- **对外接口契约**：[modules/l2_tools.md](modules/l2_tools.md)

### 14. FileTool

把 L1 FileSystem 能力包装成 agent 工具的适配层 + 路径权限域 + 越界守护（L2 agent 语义）。

- **装配归属**：按需（任何允许 agent 操作文件系统的 daemon 装）
- **资源**：无（thin wrapper / 路径权限策略由调用方配置）
- **依赖**：FileSystem (L1) / Tools (L2 / Tool 协议类型)
- **耦合**：无
- **对外能力**：read / write / search / ls 工具定义对象（实现 Tool 协议）/ 路径权限域配置 / 越界守护
- **导出工具**：read, write, search, ls
- **被谁调用**：Assembly（装配期注册到 Tools registry）
- **对外接口契约**：[modules/l2_file_tool.md](modules/l2_file_tool.md)

### 15. ShellTool

把 L1 ProcessExec 能力包装成 agent 工具的适配层 + 命令调用约束（L2 agent 语义）。

- **装配归属**：按需（任何允许 agent 调用外部命令的 daemon 装）
- **资源**：无（thin wrapper）
- **依赖**：ProcessExec (L1) / Tools (L2 / Tool 协议类型)
- **耦合**：caller 自负 shell mode 跨 OS 风险（应然 ProcessExec 不暴露 shell mode）
- **对外能力**：exec 工具定义对象（实现 Tool 协议）/ 可选命令白/黑名单
- **导出工具**：exec
- **被谁调用**：Assembly（装配期注册到 Tools registry）
- **对外接口契约**：[modules/l2_shell_tool.md](modules/l2_shell_tool.md)

## L3 执行与连接

### 16. StepExecutor

单步执行器。调一次 LLM / 如果返回 tool_use 则执行对应 handler / 返回更新后的 messages + 停止信号。

- **装配归属**：按需（随 AgentExecutor 装配）
- **资源**：无
- **依赖**：LLMService (L1)
- **耦合**：工具 handler 协议（装配方注入工具 map）/ 执行过程事件回调（由调用方透传）
- **定义的协议**：`ToolHandler`（工具调用协议）
- **对外能力**：单次 LLM 调用 / tool_use 分组并行执行 / messages in-place 追加 / max_tokens 截断修复 / context window 识别 / 停止信号
- **唯一消费者**：AgentExecutor
- **对外接口契约**：[modules/l3_step_executor.md](modules/l3_step_executor.md)

### 17. AgentExecutor

驱动一次完整的 agent 执行。反复调 StepExecutor 跑单步 / 每次 LLM 调用后通过 SessionStore 落盘 / 直到收到停止信号。

- **装配归属**：按需
- **资源**：无
- **依赖**：StepExecutor, SessionStore
- **耦合**：执行过程事件回调（由调用方透传）/ abort 回调（由 Daemon 注入 / 用于外部中断）
- **定义的协议**：`AbortSignal`（中断协议）
- **对外能力**：完整 agent 执行循环 / 跨步计数与熔断 / 每步完成后落盘 / maxSteps 守卫 / 每步钩子供调用方注入 step_yield 判定
- **被谁调用**：Runtime（常驻 agent 执行）/ SubAgent（一次性子代理）
- **对外接口契约**：[modules/l3_agent_executor.md](modules/l3_agent_executor.md)

### 18. SubAgent

一次性 agent 执行原语 — 跑一次完整的 react loop 生命周期。**纯执行原语 / 不管外部排队、并发、崩溃恢复**（那些是 L4 任务调度方的事）。与 StepExecutor / AgentExecutor 同层 / 都是「agent 执行相关的 L3 原语」。

- **装配归属**：按需（由调用方实例化）
- **资源**：无自有资源 / 执行期消费调用方提供的 input + output channel
- **依赖**：LLMService (L1) / SkillSystem (L2) / Tools (L2 / 下向依赖)
- **耦合**：执行过程事件回调（由调用方透传）
- **定义的协议**：无（承接 AgentExecutor 的 StreamCallbacks / AbortSignal）
- **对外能力**：单次任务消费（prompt + 可选父上下文 messages + tools + timeouts + systemPrompt）/ 父代理上下文快照继承 / 生命周期事件审计 / 总超时与 idle 超时
- **被谁调用**：L4 任务调度方（当前 TaskSystem 是唯一 caller / 应然作为 task executor 协议实现者）
- **对外接口契约**：[modules/l3_subagent.md](modules/l3_subagent.md)

### 19. Gateway

【设计中 / 先不实现】管理外部客户端与系统之间的实时交互（数据面）。订阅 Stream 推送事件给客户端 / 接收客户端信号路由到系统内部 / 提供 `ask_user` 工具让 agent 阻塞等待用户回复。与 CLI（控制面 / daemon 生命周期管理）并列 / 共同构成系统对外边界。

- **装配归属**：按需（任何需要把 agent 与外部客户端实时交互桥接的 daemon 装 / 依赖 Transport 装配）
- **资源**：无（派生运行时状态：连接表 / pending 询问 / 中断防抖时戳 / 重启从事件流重建）
- **依赖**：Stream（通过工厂注入）/ Transport（可选 / L1 / 无则进入 offline mode）
- **耦合**：定义中断回调协议 / 只读订阅 Stream / Transport 生命周期绑定 / 连接视图派生
- **导出工具**：ask_user（向用户提问等待回复）
- **被谁调用**：Daemon（启动时创建）
- **对外接口契约**：[modules/l3_gateway.md](modules/l3_gateway.md)

## L4 任务与业务

### 20. TaskSystem

通用异步任务调度 + 崩溃恢复 + 结果持久化回传。外部调用方直写 `tasks/pending/`；TaskSystem 内 FileWatcher 订阅 pending/ 拾起 → 状态机流转 → 派发到对应 task type 的 executor → 回传父 claw。**应然为 generic task queue / 不绑死单一 task type**；当前实然只装 subagent task type。

- **装配归属**：按需
- **资源**：`tasks/(pending/running/done/failed/results/)` 五子目录
- **依赖**：FileSystem / LLMService / AuditLog / Stream / FileWatcher / SkillSystem / OutboxWriter（Messaging）/ 调用 L3 执行原语
- **耦合**：可选父级流式输出 sink（运行期注入）
- **导出工具**：spawn / dispatch（ask_motion 为 dispatch 内部实现细节 / 不对外暴露）
- **对外能力**：subagent 任务调度（fs 驱动）/ tool 任务调度（内存驱动）/ 崩溃恢复 / 大结果持久化 + 已发送 marker 幂等 / 取消 / 优雅 shutdown / gitignore 模式常量
- **被谁调用**：Runtime / Assembly / cron / daemon
- **对外接口契约**：[modules/l4_task_system.md](modules/l4_task_system.md)

### 21. ContractSystem

契约的完整生命周期管理。创建 / 状态追踪 / 验收判定（脚本/LLM）/ 重试+升级 / 暂停+恢复+取消 / 归档。

- **装配归属**：按需
- **资源**：`contract/active/` + `contract/paused/` + `contract/archive/`（每契约子目录含 `progress.json` + `progress.lock`）+ `by-contract/` 索引
- **依赖**：FileSystem / AuditLog / Messaging / ProcessExec / TaskSystem
- **耦合**：定义通知回调协议供装配方注入 / progress.lock 文件锁
- **定义的协议**：`contract_completed` 事件发布协议 / `onNotify` 回调协议
- **导出工具**：done（pause/resume/cancel 走 CLI）
- **对外能力**：契约 CRUD / subtask 状态迁移 / 脚本验收 / LLM 验收（经 TaskSystem 调度 verifier 子代理）/ 重试 + escalation / 文件锁并发保护 / 完结归档 / 验收结果 inbox 反写
- **被谁调用**：Runtime / agent 工具层（done）/ CLI
- **对外接口契约**：[modules/l4_contract_system.md](modules/l4_contract_system.md)

## L5 外壳与能力

### 22. Runtime

常驻 agent 的事件驱动循环。等待事件 → 启动一轮 AgentExecutor 执行 → 执行完成后回到等待。Runtime 不做装配（已独立为 Assembly L6c）/ 只接收装配好的 instances 跑循环。Motion 和 Claw 的差异由 Assembly 按 identity 配置分支决定 / Runtime 内部无身份分支。

- **装配归属**：按需（任何运行常驻 agent 的 daemon 装）
- **资源**：无
- **依赖**：Assembly 预制后注入的依赖集合（13 port interface / 通过 RuntimeDependencies 接收）
- **耦合**：执行过程事件回调（publisher-subscriber 形态 B / 由 Assembly 注入透传）
- **内部组件**：`ContextInjector`（构建 system prompt 的组装器）/ `DispatchTool`（dispatch 工具实现 / phase335 物理迁入 Runtime 子模块）
- **对外能力**：常驻事件循环驱动 / session 生命周期 / turn 级 audit / 三种 turn 中断响应 / snapshot 轮级 commit / TaskSystem 生命周期协调 / 用户中断 / motion 身份注入
- **被谁调用**：Daemon
- **对外接口契约**：[modules/l5_runtime.md](modules/l5_runtime.md)

### 23. Cron

定时调度能力。按时间表（daily / hourly / interval / 支持 500ms 颗粒度）触发任务处理器；同时窗内同任务仅触发一次；同任务不重叠执行；处理器异常隔离。

- **装配归属**：按需（任何 daemon 装配方按 cron 配置启用决定是否装配）
- **资源**：无磁盘资源 / 维护调度计时器 + 去重时窗状态 + 并发状态
- **依赖**：无产品模块依赖（应然「定时调度框架 / jobs 由装配方注入」）
- **耦合**：定义任务注入协议 / 由装配方聚合任务清单后提供
- **定义的协议**：`CronJob`（任务注入协议 / 含名称 + 启用开关 + 时间表 + 处理器；处理器异常由本模块隔离）
- **对外能力**：调度启停 / 单次 tick 触发 / 时间表解析
- **被谁调用**：Assembly（装配 + 启动 / 具体任务业务语义归各业务模块 / 装配方聚合后注入）
- **对外接口契约**：[modules/l5_cron.md](modules/l5_cron.md)

### 24. MemorySystem

【设计中 / 先不实现】智能体的持久化记忆能力。dream（经验提炼 / 知识沉淀）+ 记忆查询能力供 agent 工具消费。

- **装配归属**：按需
- **资源**：每 claw 的 deep dream 状态文件 / 跨 claw 的 random dream 状态文件（motion 持有 / 整合 scope 专用）
- **依赖**：LLMService / TaskSystem（dream 子代理调度 / 应然待修订 design-gap）/ Messaging（结果投递）/ FileSystem
- **耦合**：无
- **对外能力**：deep dream 触发（per claw 经验提炼）/ random dream 触发（跨 claw 整合 / motion scope）/ 记忆查询能力供工具层消费
- **导出工具**：memory_search
- **被谁调用**：Assembly / Cron 的 dream-trigger 任务
- **对外接口契约**：[modules/l5_memory_system.md](modules/l5_memory_system.md)

### 25. ContractRetro

实现 Philosophy「各个 claw 的特长能力可以轻易复制过来」的核心机制。订阅 contract 完成事件 / 提炼 claw 在此契约中展现的特长能力 / 沉淀进自有的能力模板库供 dispatch 工具读取装配给其他 claw。

- **装配归属**：按需
- **资源**：能力模板库目录（per ContractRetro 实例自有 / 存储提炼出的技能模板 / dispatch 工具经 ContractRetro 对外入口读取）
- **依赖**：TaskSystem（派发 retro subagent）/ ContractSystem（订阅 contract_completed 事件）
- **耦合**：订阅 ContractSystem 发布的 `contract_completed` 事件协议
- **对外能力**：retro 触发（事件驱动）/ retro prompt 构造 / 能力提炼 / 技能模板沉淀 / 模板查询（供 dispatch 读）
- **被谁调用**：Assembly（装配 + 订阅 contract_completed 事件）/ dispatch 工具（读能力模板库）
- **对外接口契约**：[modules/l5_contract_retro.md](modules/l5_contract_retro.md)

## L6 入口

L6 分三类：
- **L6a 进程入口**（Daemon、Watchdog）：由 OS / shell / systemd 直启的 main 进程
- **L6b 交互入口**（CLI）：外部对系统的操作通过 CLI 唯一入口
- **L6c 装配**（Assembly）：模块装配根 / 由 L6a 进程入口调用 / 按 identity 配置分支决定启哪些模块 + 拓扑 + 注入跨模块回调

### 26. Daemon

进程生命周期管理。main 入口 / 信号处理 / 按 Assembly 返回的 Instances 触发 shutdown。装配职责已独立为 Assembly 模块（L6c）/ Daemon 不知道被启的模块内部细节 / 只对 Instances 调反向清理。

- **装配归属**：按需（任何 long-running daemon 进程入口装）
- **资源**：process signal handler（SIGTERM/SIGINT/uncaughtException/unhandledRejection）
- **依赖**：Assembly（通过 Instances 接收装配好的模块）/ Runtime 公共 API 消费仅 3 方法（处理一批 / 重试上一轮 / 中断）
- **耦合**：无
- **对外能力**：进程主入口 / 启动期装配 + snapshot commit / 事件循环驱动 / inbox 阻塞等待 / 信号 handler / 异常退出审计 / 优雅关停
- **被谁调用**：进程入口（daemon-entry 脚本）
- **对外接口契约**：[modules/l6_daemon.md](modules/l6_daemon.md)

### 27. CLI

系统的标准操作接口。用户和智能体通过同一个接口管理系统。Motion 作为用户的代理 / 通过 exec 执行 CLI 命令管理 clawforum。

- **装配归属**：命令进程（短生命周期 / 每次 invoke 独立 / 不装配在 daemon 内部）/ 同时提供 status 工具给 agent 进程内调用
- **资源**：无
- **依赖**：L1-L5 各模块（每命令独立实例化所需模块子集 / 依赖方向满足 L6b → L1-L5 下向约束）
- **耦合**：无
- **对外能力**：daemon 生命周期管理（start/stop/status/init）/ contract 操作（create/pause/resume/cancel/log）/ chat-viewport TUI / claw / motion / watchdog 子命令
- **导出工具**：status（进程内调用 CLI 命令处理函数 / 不 spawn 子进程）
- **被谁调用**：用户终端 / 智能体（通过 exec 工具）

### 28. Watchdog

进程级健康监控。独立进程 / 观察 + 干预系统健康状态。非智能体（无 LLM / 无 prompt）/ 是监督基础设施。

- Motion 存活监控与自动重启（指数回避 backoff / 上限 5 min）
- Claw 崩溃检测（was alive → now dead 且持合约）→ drop `crash_notification` 文件到 motion inbox
- Claw 不活跃检测（有活跃契约但 LLM 事件 timeout）→ drop 提醒到 motion inbox

- **装配归属**：独立进程（不装进 motion / claw daemon / 由 CLI `watchdog start` 派生独立运行）
- **资源**：watchdog.pid / watchdog-state.json / logs/watchdog.log / audit.tsv（复用全局）
- **依赖**：ProcessManager / AuditLog / FileSystem / CLI config
- **耦合**：motion 作为 claw crash 通知中介
- **对外能力**：进程存活轮询 / motion 自动重启 + backoff / claw 崩溃检测 + motion 中介通知 / claw 不活跃提醒 / 系统 liveness 审计 / start/stop/status 命令
- **被谁调用**：CLI `watchdog start/stop/status` 命令
- **对外接口契约**：[modules/l6_watchdog.md](modules/l6_watchdog.md)

### 29. Assembly

模块装配根。按 identity 配置分支决定启哪些模块 + 调各模块 setup 函数 + 注入跨模块回调 + 返回 Instances 句柄集。Assembly 会随模块数量增加而变大 / 但对外耦合界面恒定为「装配 + 反向清理」两动作 / 外部模块增减不影响调用方。

- 依赖拓扑：读 config / 按 L1 → L5 顺序构造各模块实例
- 跨模块回调注入：执行过程事件回调 / interrupt / Cron handler / LLMEventSink fan-out / TransportErrorEvent fan-out 全部在此接入

- **装配归属**：按需（任何 daemon 进程入口需要装配模块图时调用）
- **资源**：无
- **依赖**：L1-L5 各模块的 setup 函数
- **耦合**：无
- **对外能力**：按 identity + config 构造 Instances 句柄集 / 跨模块回调注入 / 装配期失败结构化审计 / 反向拓扑关停 / lockfile 冲突识别
- **被谁调用**：Daemon（进程入口间接调用）
- **对外接口契约**：[modules/l6_assembly.md](modules/l6_assembly.md)

## 未来演进方向

### 目录驱动化

当前工具直接调用业务模块 API。未来可能演进为"工具只写目录、业务模块轮询响应"。为保留这个可能性 / V1 实现遵守以下自律：

1. **工具调业务模块只做"下单"动作 / 不消费返回值** — spawn 立即返回 taskId / 不等任务结果；done 只做"标记"动作。工具不持有 handle / 不订阅后续事件
2. **业务模块把磁盘当权威状态 / 不在内存里缓存"真相"** — TaskRunner 的待跑任务从目录读 / ContractSystem 的活跃契约从 contract/active/ 读
3. **所有写操作用原子写** — 临时文件 + rename / 未来工具直接写目录时格式完全一致
4. **文件格式严格定义** — 跨模块数据结构集中定义 / 业务模块写出的文件 = 未来工具要写出的文件 / 同一份 schema 共用

### 分布式部署

当前所有模块在单机运行。未来跨主机部署时：

- CLI 命令通过 SSH 执行目标主机的 CLI / 调用点不变
- Transport 内部换成跨网络协议 / 对外接口不变
- Watchdog 的健康检查通过 CLI 远程执行 / 不直接读目标主机文件

### 跨 OS 支持

当前实现假设 POSIX（macOS / Linux）。未来支持 Windows / 其他 OS 时 / 差异由 L1 / L2 抽象层吸收 / 上层业务模块（L2+ 业务模块 + L3-L6）对 OS 平台无感知。

需改的模块（OS 抽象集中在 L1）：

- **ProcessExec（L1）**：进程 OS 能力的统一 wrapper — exec / detached spawn / kill / 存活检查 / 进程查找等所有 OS 进程操作 / 内部按 OS 分支（POSIX child_process + signals + pgrep / Windows TerminateProcess + tasklist 等）— 跨 OS 主战场
- **Transport（L1）**：当前 `UnixDomainSocketTransport` 绑死 POSIX / Windows 需 Named Pipe 实现 / 协议层切换（已是 Transport 抽象的设计目标 / 实现层补 NamedPipeTransport）
- **FileSystem（L1）**：clawforum 根路径约定（`~/.clawforum` Unix vs `%APPDATA%\clawforum` Windows）/ 路径常量需 OS 分支
- **FileWatcher（L1）**：chokidar 已 wrap 多平台差异（inotify / FSEvents / ReadDirectoryChangesW）/ 改动量极小 / 主要测试覆盖
- **CLI / TUI（L6b）**：终端 ANSI / TTY 行为 / Windows cmd / PowerShell 表现差异 — 需测试覆盖

不需改的模块：所有 L2+ 业务模块（ProcessManager / Stream / SessionStore / AuditLog / Messaging / Snapshot / SkillSystem / Tools / FileTool / ShellTool / TaskSystem / ContractSystem / MemorySystem / Cron / SubAgent / StepExecutor / AgentExecutor / Gateway / Runtime / Daemon / Watchdog / Assembly / ContractRetro）— 通过 L1 抽象访问 OS 能力 / 不直接 import OS-specific stdlib。注意：ProcessManager 应然不直接 import `child_process` / 通过 ProcessExec 调。

跟「分布式部署」同型架构原则：**L1 / L2 抽象层吸收部署 + OS 差异 / 上层业务模块对部署模式 + OS 平台无感知**。

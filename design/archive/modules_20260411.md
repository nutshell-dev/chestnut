# Clawforum 模块清单（重构草案）

共 17 个模块，按依赖方向由下往上排列。每个模块标注一句职责与直接依赖。

## 基础设施层

### 1. FileSystem
文件读写与权限检查的统一入口。对上层屏蔽 fs.promises，封装原子写、目录创建、路径越界检查。
- **依赖**：无

### 2. LLMService
LLM 调用的统一服务。负责 provider adapter 管理、请求组装、KV cache 标记、重试与超时。
- **依赖**：无（HTTP 为外部依赖）

### 3. MessageCodec
inbox/outbox 消息的序列化与反序列化。frontmatter + body 的唯一编解码点。
- **依赖**：无

## 持久化层

### 4. SessionStore
对话历史的持久化。管理 `session/current.json` 与 `session/archive/`，支持断点恢复。
- **依赖**：FileSystem

### 5. Recording
流式事件与审计记录的写入。stream.jsonl（TUI 实时查看）、steps.jsonl（步骤记录）、audit.tsv（审计）。
- **依赖**：FileSystem

### 6. Messaging
inbox/outbox 的目录结构、消息投递与拉取。所有跨 agent 通信必经之路。
- **依赖**：FileSystem, MessageCodec

## 执行层

### 7. ReActEngine
纯 ReAct 循环引擎。输入 messages + tools，输出下一步动作；不做 I/O，不知道"任务"概念。
- **依赖**：LLMService

### 8. ToolSystem
工具注册、profile 管理与执行调度。内建工具（read/write/exec/spawn/dispatch/...）集中维护。
- **依赖**：FileSystem, Messaging（个别工具另有依赖，如 spawn→TaskRunner）

### 9. AgentExecutor
一次性 agent 执行器。给定 prompt + tools + config，跑一次 ReAct 循环，返回 Result。所有一次性 agent（Dispatcher/Miner/Describer/Verifier/Dream/Spawn subagent）共用此执行器。
- **依赖**：ReActEngine, ToolSystem, SessionStore, Recording

### 10. TaskRunner
任务生命周期管理器。接收 TaskSpec，建任务目录，调 AgentExecutor 跑任务，管结果回传与崩溃恢复。tasks 目录唯一写入者。
- **依赖**：AgentExecutor, Messaging, FileSystem

## 业务主干

### 11. ContractSystem
契约生命周期管理。管 `contract/active/`、`contract/completed/` 目录与状态机，通过 TaskRunner 调度 verifier agent 做异步验收。
- **依赖**：TaskRunner, Messaging, FileSystem

## 长驻 agent 外壳

### 12. Runtime
长驻 agent 的外壳。inbox 驱动循环，每次有消息就调 AgentExecutor 跑一轮，处理工具结果与契约事件。Claw、Motion 都是 Runtime + 不同配置。
- **依赖**：AgentExecutor, Messaging, SessionStore, ContractSystem, TaskRunner, Recording

## 工具能力

### 13. SkillSystem
用户定义 skill 的目录扫描与按名读取。启动期扫描 `skills/` 构建 catalog（name + description），运行期提供 `list()` 和 `readSkill(name)` 读出 SKILL.md 正文。skill 的执行体在正文里作为提示词片段被智能体消费，SkillSystem 本身不执行 skill、不拼 prompt、不管 references/ 子目录（由智能体按需 read）。
- **依赖**：FileSystem
- **SkillTool**：作为 Daemon 装配的 bridge 工具（不是 ToolSystem 内建），构造时注入 SkillSystem 引用。工具描述里带 catalog（name + desc），让智能体"看得到"有哪些 skill；调用时按 name 读出正文返回。ToolSystem 不依赖 SkillSystem——这层关系在 Daemon 装配时一次性连好。

## 后台组件

### 14. Cron
周期性任务触发器。内建 jobs：deep-dream、random-dream、disk-monitor、llm-stats。dream jobs 通过 TaskRunner 启动一次性 agent。响应式跨 agent 观察**不归 Cron**——那是 PeerWatcher 的事。
- **依赖**：TaskRunner, FileSystem

### 15. PeerWatcher
跨 agent 响应式观察组件。在本 agent 启动时监视 peer agent 的特定目录（当前：claw 的 outbox），检测到变化时通过 Messaging 投递汇总通知到自己的 inbox。使用 fs.watch + periodic fallback poll 保证不漏事件。启用与否取决于 agent 配置——Motion 启用（监视所有 claw outbox），Claw 不启用（claw 之间互不观察）。

**动态 peer 发现**：PeerWatcher 同时监视 clawforum agents 父目录，检测新 claw 目录出现 / 老 claw 目录消失，动态增减对应的 outbox watcher。periodic fallback poll 同时 re-list 父目录，防止 fs.watch 漏检新 claw。这不违反"启动期拓扑凝固"——拓扑凝固约束的是 module class 级依赖图，而内部 watcher 集合属于运行时状态，允许随 peer 增减变化。
- **依赖**：FileSystem, Messaging

## 入口

### 16. Daemon
事件循环与中断处理。驱动 Runtime 主循环，处理 heartbeat、startup check、signal。
- **依赖**：Runtime, Messaging, FileSystem

### 17. CLI
命令行入口。clawforum claw/motion 等命令的解析与分发。同时是跨 agent on-demand 读取的触发入口（Motion agent 通过 exec 工具调用 CLI 命令读取 peer agent 状态）。
- **依赖**：Daemon, Runtime, FileSystem

## 依赖关系速览（DAG）

```
CLI / Daemon
     ↓
  Runtime ──────────────────┐
     ↓                      │
ContractSystem    Cron      │
     ↓             ↓        │
  TaskRunner ──────┘        │
     ↓                      │
AgentExecutor ──────────────┤
     ↓                      ↓
ReActEngine ← ── ── ── ToolSystem ← ── SkillSystem
     ↓                      ↓
 LLMService      SessionStore  Recording  Messaging
                        ↓           ↓          ↓
                   FileSystem  FileSystem  FileSystem / MessageCodec
```

**PeerWatcher（旁路背景组件，不在主 DAG 链路上）**

```
PeerWatcher ──→ FileSystem  (watch peer 目录 + fallback poll)
            └─→ Messaging    (投递通知到自己 inbox)
```

PeerWatcher 由 Daemon 在启动时拉起，与 Runtime / daemon-loop / Cron 并行运行。它只向下依赖 FileSystem 和 Messaging，**不经过** Runtime / TaskRunner / AgentExecutor / ToolSystem 等任何执行链路的模块——它本质上是一个独立的观察循环，跟 agent 的 ReAct 执行路径完全解耦。不把它画进主 DAG 是为了避免产生"Runtime → PeerWatcher → ToolSystem"这种假拓扑错觉。

- **无反向依赖**：任何模块都不会调用比自己层级高的模块
- **SubAgent 不存在**：被 AgentExecutor 取代
- **MotionRuntime 不存在**：Motion 是 Runtime 的一种配置
- **ToolTask 路径删除**：工具调用一律同步，耗时工作走 spawn→AgentExecutor

## 设计自律：为未来"目录驱动化"保留可能性

当前模块设计不采用"工具只写目录、业务模块轮询响应"的方向（方向 C），工具仍然直接调用业务模块 API。但为了未来可以无痛切换到 C，V1 实现要遵守以下四条自律：

1. **工具调业务模块只做"下单"动作，不消费返回值**
   - spawn 立即返回 taskId，不等任务结果；done 只做"标记"动作，不返回详细状态对象
   - 工具不持有 handle、不订阅后续事件——一次调用就结束
   - 这样未来把调用改成"写目录"时，工具侧无需改动调用点

2. **业务模块把磁盘当权威状态，不在内存里缓存"真相"**
   - TaskRunner 的"有哪些待跑任务"从目录读，不维护内存 pending map
   - ContractSystem 的"当前活跃契约"从 `contract/active/` 读，不维护内存集合
   - 内存里可以有"正在跑"的运行期状态（比如 child process handle），但"下一步该做什么"的判定必须回到磁盘
   - 未来切 C 时，业务模块的扫描逻辑已经就位，只是触发源从 API 调用改为文件出现

3. **所有写操作从一开始就用原子写**
   - 业务模块内部写状态文件一律走 FileSystem 的原子写原语（临时文件 + rename）
   - 未来工具直接写目录时，是 1:1 替换业务模块内部的写点，格式完全一致

4. **文件格式从一开始就严格定义**
   - TaskSpec、ContractMarker、InboxEnvelope 等跨模块数据结构集中在一处定义（比如 `task-types.ts` / `contract-types.ts`）
   - 业务模块写出的文件 = 未来工具要写出的文件，同一份 schema 两侧共用
   - 避免"业务模块随手拼 JSON 写盘"这种私有格式

**明确不做的事（V1 阶段）：**
- 不引入 tick() 扫描机制，业务模块不开后台轮询
- Runtime 主循环不调任何 tick
- 工具依然直接 `await taskRunner.enqueue(spec)` 这类 API 调用
- 不搞"已处理"标记机制

## 跨 agent 观察：两条互补的路径

Design Principle "多个智能体的信息不隔绝" 暗含一类**跨 agent 读**动作——Motion 要能看到其他 claw 的 outbox、status、contract 等。当前代码里的实例是 `daemon-loop.ts` 里 Motion daemon 轮询 `scanClawOutboxes()`。新设计把这件事拆成两条互补的路径，分别处理"通知触达"和"详情获取"。

### 路径 A：PeerWatcher（push 通知，响应式）

**什么时候用**：Motion 需要在 claw outbox 有变化时**及时**得到提醒，但不需要立即看到详情。

**怎么做**：PeerWatcher 模块在 Motion 进程启动时被拉起，用 fs.watch（+ fallback poll）监视所有 claw 的 outbox 目录。检测到变化就汇总当前未读状态，通过 Messaging 把一条提醒消息投进 Motion 自己的 inbox，使用 `replaceKey='peer:claw-unread'` 去重（pending 中已有的被新的覆盖）。

**产出**：Motion inbox 里的一条提醒消息，内容包含"哪些 claw 有未读、各自多少条"。Motion agent 通过 ReAct 循环正常消费这条消息，决定下一步动作。

### 路径 B：CLI + exec（pull 详情，on-demand）

**什么时候用**：Motion agent 看到 PeerWatcher 的提醒后，决定要查看某个 claw 的**具体内容**。

**怎么做**：Motion agent 在 ReAct 循环里用 exec 工具运行 `clawforum claw outbox <clawId>` 之类的 CLI 命令。CLI 命令直接读取 peer agent 的目录，把结果返回给 Motion agent 作为 tool_result。

**产出**：详细的 outbox 内容作为 tool_result 进入 Motion 的 messages 流。

### 两条路径的分工与约束

| 维度 | PeerWatcher (A) | CLI + exec (B) |
|---|---|---|
| 触发源 | 文件系统变化事件 | Motion agent 的 ReAct 决策 |
| 方向 | push（写 Motion inbox） | pull（Motion 主动 exec） |
| 粒度 | 汇总（有几个 claw、各自多少条） | 详情（具体某个 claw 的完整 outbox） |
| 频率 | 事件驱动，不定频 | agent 主动，按需 |
| 作用域 | Motion 进程内的背景组件 | Motion agent 的工具调用 |

**为什么不是 cron**：响应式观察必须在 claw outbox 变化时尽快触发，cron 的周期性机制引入不必要的延迟和空转开销。cron 的语义是"纯时间驱动"，跟"响应外部变化"语义不匹配。

**为什么不是 claw push**：Claw 不应该知道 Motion 的存在或位置，否则引入 N×M 耦合和"所有 claw 都要维护 motion 列表"的负担。观察的利益方是 Motion，所以由 Motion 进程负责观察。

**为什么不是 daemon-loop 内扫描**：daemon-loop 核心保持对所有 agent 类型对称（无 isMotion 分支），让 Motion 和 Claw 的 Runtime 完全统一。PeerWatcher 作为独立模块被 Daemon 启动时拉起，与 daemon-loop 并行运行但互不纠缠。

**未来分布式**：PeerWatcher 的 fs.watch 实现本地场景；跨主机时 PeerWatcher 内部换成 RPC 订阅/SSH 拉取等 transport，对外接口不变。CLI 命令同理——跨主机时 CLI 的 exec 通过 SSH 执行目标主机的 CLI，Motion 和 CLI 的调用点都不用改。

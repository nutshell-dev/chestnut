# ADR 0005 — 治理冲刺 r17-r29 方法论总结

**状态**：已采纳（2026-04-25）
**决策者**：用户 + 主会话协作演化
**来源**：r17-r29（2026-04-19 ~ 2026-04-25）13 轮治理冲刺

## Context

clawforum 项目从 r17 开始进入系统性治理冲刺阶段，目标是让 modules.md 的应然架构在代码中落地。13 轮（r17-r29）建立了完整的治理工作流，包括多 fork 并行分派、drift 管理、假合入防线、过度治理识别等方法论。

本 ADR 总结这些方法论的核心要素，为 r31+ Stage 2（模块内部并行重构）和未来项目的治理工作提供参考。

## Decision

### 1. 多 fork 并行分派模型

**结构**：每轮（r）包含 1 主会话（A）+ 2-5 fork（B/C/D/E/F）。

| 角色 | 职责 |
|---|---|
| **主会话 A（Meta）** | 起草分发表 / 硬化 feedback / 写收官报告 / ack fork 报告 |
| **fork B-F** | 按 7 节计划执行具体 phase / Step 1 Path #1 核 / 产报告 |

**分派节奏**：
- **freeze 节奏**（r29 起采用）：不强凑 5 fork / 按实际工作量分 2-4 fork / E/F 允许空
- **Meta 节奏**：每 3-5 轮 1 次 / 触发条件不满足时跳过（r30 首次跳过）

### 2. Phase 协议

**phase 号 claim**：`mkdir "coding plan/phase<N>"` 原子保证 / 不带 -p / 撞号顺延。

**7 节计划结构**（每个 step 必须）：
1. 当前状态
2. 目标
3. 修改文件
4. 设计细节
5. 不做
6. 验收
7. 风险

**phase 总览 v5 元规范**：
- 思考预算分配（总览解 phase 级 / 执行计划解 Step 级）
- 长度 ∝ 新颖度（重复性 phase 简化）
- 含 Step 间依赖关系

### 3. 修订完整性自检纪律组（7 件套）

主会话信息综合的完整性防线，覆盖 7 个对象级别（详见 ADR 0001）。

核心原则：**凭推测下结论 → 必先 Read/grep/git log 实然核**。

### 4. Drift 管理方法论

**drift = 契约/清单/登记 与实然代码的状态偏差**。

**6 种形态**（详见 feedback_verify_facts_before_plan.md）：

| 形态 | 对象 | scope |
|---|---|---|
| 1-4 | 单一事实 | Step 计划内 |
| 5a | backfill drift 核 | 契约 §7 条目 |
| 5b | 代码 phase drift 核 | 整理债 / 清单项 |
| 5c | 零结果判定 | drift 核返回 0 |
| 5d | 决策性反转必通知 | 方向反转 |
| 5e | 前提断言核 | phase 起步 |
| 5f | 登记自 drift | 登记后的 main 推进期 |

**关键洞察**：drift 登记不是一次性完成，它本身会 drift（5f 形态）。需周期性复核。

### 5. 假合入防线

fork 完成后用户未 ff 到 main / 主会话信任 fork 报告 SHA 直接陈述（详见 ADR 0002）。

**防线**：每条 SHA 必 `git branch --contains` 核 / 收官表含 ✅/❌ 列。

### 6. 过度治理识别

自动化工具暴露 ≠ 实然违规（详见 ADR 0003）。

**选轨 2**：脚本暴露作为候选 → Path #1 二次核 → 仅真违规修复。

### 7. Path #6 + Path #8 协同模板

fork 起步 Step 1 Path #1 核发现分发表假设错 → Path #6 中断 → 主会话承认 + Path #8 决策 → 改 scope 产设计文档 / 反哺下轮。

### 8. 治理度量演进

| 维度 | r17 起 | r25 起 | r30 末预期 |
|---|---|---|---|
| modules.md 应然 | 部分模块实现 | 26/27 | 27/27（100%）|
| §7.A active | ~30 | 4 + W-tier 2 | 0 + W-tier 2 |
| §7.B 用户表 | ~40 | 10+ | 7 已修 + 3 保留 |
| 整理债 | ~15 项 | C 类 4/5 | C 类 5/5（全闭环）|
| 修订完整性 | 无 | 5 件套 | 7 件套 |

## Consequence

### 正面

- **Stage 1 真完结**：r30 末 modules.md 应然 100% / Stage 2 可启动
- **方法论可复用**：治理工作流可直接应用于 Stage 2 模块内部重构
- **错误谱系可追溯**：每条方法论都有实证触发历史 / 新会话可查阅避免重犯

### 负面

- **文档维护成本**：feedback memory + ADR + 契约 三级文档需保持同步
- **方法论过度应用风险**：Stage 2 是模块内部重构 / scope 更小 / 不需要全部方法论

### Stage 2 应用建议

- **保留**：7 节计划 / phase 号 claim / 修订完整性 7 件套 / drift 管理
- **简化**：fork 数可减少（每模块独立 / 不需 5 fork） / 总览可简化
- **按需**：假合入防线（仍需）/ 过度治理识别（Stage 2 scope 小 / 风险低）

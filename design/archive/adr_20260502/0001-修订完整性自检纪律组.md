# ADR 0001 — 修订完整性自检纪律组（7 件套）

**状态**：已采纳（2026-04-25）
**决策者**：用户 + 主会话协作硬化
**来源**：r24-r29（2026-04-25）跨 6 轮主会话 / feedback_verify_facts_before_plan.md

## Context

clawforum 项目采用多 fork 并行分派 + 主会话统筹的工作流。主会话负责综合各 fork 报告、写收官报告、起草分发表。在 r24-r29 期间，主会话连续出现「凭推测下结论」的错误，具体表现为：

1. **r24 收官 over-claim**：凭「8 模块全清零」推断「全 §7.A 清零」（实然 active = 4 / 清零率 87% 非 97%）
2. **r25 末推测错**：凭上下文片段推测「14 条违规主体可能是 import type」（实然 14 全 A 类 value import）
3. **r26 D 汇报错**：凭脚本数字陈述「7 处 Tools L3→L4/L5 大头」（实然 0 真违规 / 脚本统计偏差）
4. **r27 B 设计错**：基于脚本 14 数字设计 γ 接口提取方案（实然 type import 全合规 / 过度治理）
5. **r25/r26 假合入**：凭 fork 报告 SHA 直接写收官「main 推进 N SHA」（实然 2 个 SHA 不在 main）

5 次实证的**共性根因**：主会话在综合信息时跳过了 Read 完整源 / grep 实然 / git log 核实，凭上下文中的片段印象或工具输出数字直接下结论。

## Decision

建立「修订完整性自检纪律组」—— 7 个完整性检查件套，覆盖主会话信息综合的全部盲点。每件套对应一个对象级别和硬要求。

### 7 件套

| # | 对象级别 | feedback 节 | 升格轮次 | 硬要求 |
|---|---|---|---|---|
| 1 | 跨契约级 | §契约状态汇总必全契约扫描 | r25 A | 写跨契约状态陈述前必扫全契约 `design/modules/*.md` |
| 2 | 单文件全文级 | §同文件 Edit 累积错误 | r25 A | 单文件 Edit 累积超过阈值必须 Read 全文确认 |
| 3 | 单清单 footer 级 | §footer/header 自检 | r25 A | 消化清单条目后同步 footer/header 计数 |
| 4 | 跨 fork 报告级 | §回复用户前必读完整源 | r26 A | 综合 fork 信息回复用户前必 Read 完整 fork 报告 |
| 5 | 脚本工具产出级 | §脚本暴露 ≠ 实然违规 | r27 A | 脚本/工具产出的违规清单必跑 Path #1 二次实然核 |
| 6 | design 方案 derive 级 | §design 方案 derive 必 Path #1 核 | r28 A | design phase 方案 derive 必逐项 Path #1 核 import 类型 |
| 7 | 合入 SHA 级 | §合入 SHA 必核 main | r29 A | 引用 SHA 声明「main 已合」必 `git log` 核实 |

### 共性结构

每个件套都包含：
- **触发条件**：什么场景必须执行检查
- **反模式**：典型错误做法（❌）
- **正确模式**：应该怎么做（✓）
- **触发历史**：哪次实证触发了硬化

### 硬化节奏

7 件套按实证自然积累硬化（每 1-2 次同根实证升格一条），不是预设框架填空。r29 A 7 件套完件后不再扩件套，除非新型主会话错位形态出现。

## Consequence

### 正面

- **主会话陈述准确性**：从 r24 的 97% over-claim 到 r29 后每条陈述有 Read/grep 锚点
- **跨轮可追溯**：每件套的触发历史形成错误谱系，新会话可查阅避免重犯
- **工具链可信度提升**：第 5-6 件套让 design 方案不盲目信任脚本输出

### 负面

- **主会话成本增加**：每条陈述前多一次 Read/grep/git log（约 5-30 秒）
- **不覆盖执行层**：7 件套聚焦主会话信息综合；执行层（fork agent）的 Path #1 由各 feedback 覆盖

### 适用范围

主会话（Meta phase / 收官报告 / 分发表起草 / fork 报告 ack）强制应用。fork agent 按需参考。

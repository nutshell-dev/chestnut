# Clawforum 模块清单

## 模块分层

| 层 | 模块 |
|---|---|
| 第一层:原语 | FileSystem, ProcessExec, LLMService, MessageCodec |
| 第二层:基础设施 | FileWatcher, SessionStore, EventStore, Messaging |
| 第三层:执行内核 | StepExecutor, AgentExecutor |
| 第四层:任务与业务 | SubagentSystem, ContractSystem |
| 第五层:外壳与能力 | Runtime, SkillSystem, Cron, MemorySystem |
| 第六层:入口 | Daemon, CLI |

---

## 第一层:原语

### 1. FileSystem

clawforum 进程内代码的所有文件 I/O 的唯一入口。原子写(临时文件+rename)、路径守护(baseDir 越界检查+symlink 解析)、权限域配置。外部子进程（经由 ProcessExec 启动）的文件访问不在此约束范围，未来靠 OS 级沙箱限制。

- **权限域**:构造参数传白名单,对 agent 身份零感知。Daemon 装配时为每个 agent 创建 trustedFs(enforcePermissions: false)和 agentFs(enforcePermissions: true)两个实例
- **导出工具**:read, write(handler 绑到 agentFs 实例)
- **不管**:文件内容解析、fs.watch、外部进程执行、目录业务语义
- **依赖**:无
- **被谁调用**:
  - daemon 内部模块(trustedFs):SessionStore, EventStore, Messaging, ContractSystem, SubagentSystem, SkillSystem, MemorySystem, FileWatcher, Runtime, Daemon
  - agent 工具层(agentFs):注册到 StepExecutor 的 read/write 工具
  - CLI 命令:自己构造 trustedFs 实例

### 2. ProcessExec

外部进程调用的唯一入口。封装 child_process.spawn,超时控制(SIGTERM → 宽限 → SIGKILL),输出大小限制。

- **单实例 + 参数化约束**:约束靠每次调用的参数(timeoutMs / maxOutputBytes / cwd / env),不靠实例区分
- **导出工具**:exec
- **不管**:命令白名单、沙箱(未来由 Daemon 在 agent 进程级设置,子进程继承)
- exec 当前能访问 contract/ 等受保护目录(是 feature,以后靠 OS 沙箱限制)
- **依赖**:无
- **被谁调用**:
  - agent 工具层:注册到 StepExecutor 的 exec 工具
  - daemon 内部模块:ContractSystem(跑 acceptance 脚本)

### 3. LLMService

LLM 调用的统一服务。provider adapter 管理、请求组装、KV cache 标记、重试与超时。

- **可多实例化**,每实例持有独立的 primary + fallback provider chain
- **provider 在装配期固定**,不做运行期路由
- **无状态**:不做跨请求 prefix diffing,session compaction 后 cache miss 是可接受代价
- **不导出工具**
- **依赖**:无(HTTP 为外部依赖)
- **唯一消费者**:StepExecutor

### 4. MessageCodec

inbox/outbox 消息的唯一编解码点。frontmatter + body 的序列化/反序列化、文件名生成、schema 格式守护。

- **纯函数**,不碰磁盘
- **type 值对 MessageCodec 透明**:只校验格式(非空、合法字符),不限定具体值。各业务模块自己导出 type 常量
- **结构化数据全在 frontmatter**,body 保持纯文本(给 LLM 看)
- **不导出工具**
- **依赖**:无
- **被谁调用**:Messaging

## 第二层:基础设施

### 5. FileWatcher

"订阅路径变化 → 收到事件"的能力。fs 原生事件不可靠时的 polling 补漏、多平台行为差异抹平。

- 不懂业务目录语义,不读文件内容,不持久化自己的状态(订阅集合是运行期状态,重启重建)
- 路径守护借 FileSystem 的路径工具
- **不导出工具**
- **依赖**:FileSystem
- **唯一的观察模块**:上层模块按需订阅,PeerWatcher 不再独立存在
- **消费者**:Runtime(监听 inbox 目录变化)、ContractSystem(监视 peer agent 的契约状态变化)

### 6. SessionStore

messages 数组的持久化读写。唯一负责把对话历史原子写入磁盘、从磁盘恢复的模块。

- **SessionStore 是 messages 数组的唯一写入者**,不管存在哪个目录
- **不绑定固定目录**:按构造时指定的 base path 工作
  - 常驻 agent:Daemon 装配时指向 `session/`
  - 一次性子代理:SubagentSystem 创建时指向 `tasks/{taskId}/`
- 不解析 messages 内容,不做 compaction/摘要
- **不导出工具**
- **依赖**:FileSystem
- **被谁调用**:AgentExecutor(每次 LLM 调用后落盘,启动时恢复)

### 7. EventStore

事件日志资源的唯一归属。写入、订阅实时流、历史读取都经过此模块。stream(实时查看)、steps(执行步骤记录)、audit(状态迁移审计)。

- 对写入内容不透明,不做日志查询/聚合/过滤
- **事件日志的唯一写入者**
- 对外暴露实时事件流订阅和历史读取入口
- **不导出工具**
- **依赖**:FileSystem
- **被谁调用**:AgentExecutor(每步执行)、Runtime(事件处理)、SubagentSystem / ContractSystem / MemorySystem(状态迁移)、CLI(通知类事件)

### 8. Messaging

inbox/outbox 的目录管理、消息投递与拉取。所有跨 agent 消息通信的必经之路。

- 管 inbox(pending/done/failed)和 outbox(pending)的目录结构与文件生命周期
- 投递时调 MessageCodec 编码,拉取时调 MessageCodec 解码
- **replaceKey 去重**:投递时如果 pending 里已有同 replaceKey 的消息,原子替换
- **inbox/outbox 目录的唯一写入者**
- 不解析消息的业务语义
- **导出工具**:send
- **依赖**:FileSystem, MessageCodec
- **被谁调用**:
  - Runtime:拉取 inbox/pending 里的消息驱动事件循环
  - SubagentSystem:交付子代理结果到调用方 agent 的 inbox
  - ContractSystem:投递验收结果/完成通知到 claw 和 Motion 的 inbox
  - MemorySystem:投递记忆整合结果到 agent 的 inbox
  - CLI 命令:投递系统通知到 target agent 的 inbox(如 contract create 后的通知)

## 第三层:执行内核

### 9. StepExecutor

单步执行器。内部持有工具 map(名字 → 定义 + handler),装配期由 Daemon 注入。调一次 LLM,如果返回 tool_use 则查找并执行对应 handler,返回更新后的 messages + 停止信号。

- **只跑一步**,不拥有循环
- 不做 messages 持久化
- 不知道 agent 身份
- **工具 registry 在此模块内部**:工具由各业务模块导出,Daemon 装配时注入。调用方通过工具名列表(数据)指定子集,StepExecutor 不关心子集的业务含义
- **cache 命中约束**:一次 agent 执行过程中,传给 LLMService 的 tools 定义和 system prompt 保持内容与顺序不变,确保 LLM 请求前缀命中 KV cache
- **不导出工具**
- **依赖**:LLMService
- **唯一消费者**:AgentExecutor

### 10. AgentExecutor

驱动一次完整的 agent 执行。反复调 StepExecutor 跑单步,每次 LLM 调用后通过 SessionStore 落盘 + EventStore 写事件,直到收到停止信号。

- 拥有循环逻辑和停止判断(max steps / LLM 返回 stop / 超时)
- **每步之间 SessionStore 落盘**,保证"任意时刻中断都合法"
- 工具名列表作为数据从调用方透传到 StepExecutor
- 不管 inbox / 契约 / 任务目录
- **不导出工具**
- **依赖**:StepExecutor, SessionStore, EventStore
- **被谁调用**:
  - Runtime:长驻 agent 每收到一批 inbox 消息,构造一次 AgentExecutor 调用
  - SubagentSystem:一次性子代理的执行

## 第四层:任务与业务

### 11. SubagentSystem

一次性子代理的唯一工厂 + 生命周期管理。配置(工具集、prompt、超时) → 创建任务目录 → 调 AgentExecutor 执行 → 状态追踪 → 结果通过 Messaging 交付给调用方 agent。

- 所有子代理身份(miner/verifier/dream/...)的工具集定义集中在此
- 任务目录(tasks/)是此模块管理的持久化资源,任务状态持久化在磁盘,不依赖内存队列
- SessionStore 由 AgentExecutor 自己构造(SubagentSystem 只传工作目录路径),符合"跨边界只传数据"
- **导出工具**:spawn, dispatch
- **不导出工具给内部**:契约创建走 CLI(exec),SubagentSystem 不依赖 ContractSystem
- **依赖**:AgentExecutor, Messaging, FileSystem, EventStore
- **被谁调用**:
  - agent 工具层:spawn(通用子代理)、dispatch(mining mode,创建 miner 子代理)
  - ContractSystem:触发 verifier 子代理做验收、契约完成后触发复盘子代理(继承 miner 上下文)
  - MemorySystem:触发 dream 子代理

### 12. ContractSystem

契约的完整生命周期管理。创建、状态追踪、验收判定(脚本/LLM)、重试/升级、暂停/恢复/取消、归档。

- 契约目录(contract/)是此模块管理的持久化资源
- 验收有两条路径:脚本验收直接调 ProcessExec,LLM 验收调 SubagentSystem 创建 verifier 子代理
- 验收结果的通知由 ContractSystem 自己通过 Messaging 发出(业务语义在此模块内闭合)
- 契约创建由 miner 子代理通过 exec 调用 CLI 完成,不经过 ContractSystem 的工具导出
- **导出工具**:completeSubtask（done）
- pause, resume, cancel 走 CLI,智能体需要时通过 exec 调用
- **依赖**:FileSystem, FileWatcher, ProcessExec, SubagentSystem, Messaging, EventStore
- **被谁调用**:
  - agent 工具层:completeSubtask(claw 报告子任务完成)、pause、resume、cancel
  - Runtime:加载活跃契约

## 第五层:外壳与能力

### 13. Runtime

常驻 agent 的事件驱动循环。收到事件 → 启动一轮 AgentExecutor 执行 → 执行完成后回到等待。Motion 和 Claw 的 Runtime 通过配置区分,核心循环统一。

- 事件来源当前是 inbox 消息,未来可能扩展
- 不识别消息类型,不做业务逻辑,只管"有事件就执行一轮"
- 一次性子代理没有 Runtime——SubagentSystem 直接调 AgentExecutor
- **不导出工具**
- **依赖**:AgentExecutor, Messaging, FileSystem, EventStore
- **被谁调用**:Daemon(启动常驻 agent)

### 14. SkillSystem

技能注册表。扫描 skills/ 目录加载技能元信息(name/description/version),渐进式披露——启动时只加载 frontmatter,调用时才加载完整内容。

- skills/ 目录是此模块管理的持久化资源
- **导出工具**:skill(列出可用技能、加载技能内容)
- **依赖**:FileSystem
- **被谁调用**:
  - agent 工具层:skill 工具
  - Daemon:装配时初始化

### 15. Cron

纯调度引擎。按 schedule 定时触发 handler,handler 在装配期由各业务模块注入。

- 支持 daily/hourly/interval 三种调度模式
- 防止同一 job 重叠执行
- 不含任何业务逻辑,不知道 handler 做什么
- **不导出工具**
- **依赖**:无(handler 装配期注入)
- **被谁调用**:Daemon(装配 jobs 并启动调度器)

### 16. MemorySystem

智能体的记忆整合。dream(空闲时自主思考)、经验提炼、知识沉淀。由 Cron 定时触发。

- 通过 SubagentSystem 创建子代理执行整合任务
- 整合结果写入记忆存储(磁盘文件)并通过 Messaging 投递到 agent inbox
- 记忆文件是此模块管理的持久化资源
- **不导出工具**
- **依赖**:SubagentSystem, Messaging, FileSystem, EventStore
- **被谁调用**:Cron(定时触发记忆整合 handler)

## 关键设计决策

1. **工具 registry 内置于 StepExecutor**:各模块导出工具定义 → Daemon 装配时注入 StepExecutor,无独立 registry 模块
2. **FileSystem 权限域**:agentFs + trustedFs,白名单由 Daemon 注入,FileSystem 对 agent 零感知
3. **FileWatcher 独立**:async I/O 和事件订阅是两种 API 形态,分开
4. **SessionStore 不绑目录**:唯一的 messages 写入者,base path 由调用方决定
5. **StepExecutor 只跑一步**:循环归 AgentExecutor,每步之间落盘,保证中断恢复
6. **SubagentSystem 合并 TaskRunner**:一次性子代理的完整生命周期(配置→创建→执行→状态→交付)在一个模块内,无中间层
7. **dispatch 和 spawn 是独立工具**:dispatch 发起 mining mode 流程(到契约创建完成即返回),spawn 创建通用子代理,不统一
8. **异步工具执行是 StepExecutor 的内部策略**:同步工具直接执行,异步工具提交后返回占位结果,结果通过 Messaging → Runtime 回到 agent
9. **契约创建走 CLI**:miner 子代理通过 exec 调用 CLI 创建契约,SubagentSystem 不依赖 ContractSystem,避免双向依赖
10. **ContractSystem 自己发通知 + 触发复盘**:验收通知由 ContractSystem 通过 Messaging 发出,契约完成后由 ContractSystem 调 SubagentSystem 创建复盘子代理。业务语义在模块内闭合,Runtime 不参与
11. **删除 PeerWatcher**:FileWatcher 是唯一的观察模块,上层模块按需订阅,不设专门的 peer 观察模块
12. **Cron 纯调度引擎**:零业务依赖,handler 装配期由各业务模块注入,Cron 不知道 handler 做什么
13. **MemorySystem 独立模块**:dream/记忆整合逻辑从 Cron jobs 剥离,归入 MemorySystem,由 Cron 定时触发
14. **配置是数据不是模块**:统一 config 文件,Daemon 装配时切片分发,每模块自持自己的切片

## 分布式改造影响面

只改三个模块:Messaging(投递 transport)、FileWatcher(远程 watch)、CLI(SSH transport)。其余不动。

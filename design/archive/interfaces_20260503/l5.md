## Runtime [capability, DI]

**生产方**：`l5_runtime`

**消费方**：

- `l6_daemon`（DI，daemon 主入口调 initialize / processBatch / processWithMessage / chat / abort / stop）
- `l6_assembly`（DI，装配期 createRuntime 工厂 + 注入 RuntimeDependencies）

**接口签名**（实然 = `class Runtime` + 13 port types DI / 不立 `Runtime` 抽象 interface）：


注：`ContractSystem` 名 align canonical（arch §22 + 表 1）。

**使用语义**：

- `initialize()` 必先调（once-only / 完成 dialog 复原 + TaskSystem.initialize 等子模块复原）
- `processBatch` / `processWithMessage` / `chat` 必在 initialize 后调
- `stop()` 释放资源 / 调后 instance 不可再用
- `abort()` 触发 UserInterrupt cause / 透 AgentExecutor onTurnInterrupted 钩子传给 caller
- `processBatch` 返本轮吸收 inbox 消息数（0 = 空轮）/ 每轮 turn 结束 snapshot commit（caller 不感知）
- 每次 LLM 调用后必透过 DialogStore 落盘 messages 数组（AgentExecutor stepCallback hook 内）/ 中断后从最后一次完整 LLM 调用恢复
- Runtime 不分 motion / claw 身份分支 / identity 经 systemPromptBuilder + identityToolFilter 注入
- DaemonStreamCallbacks 扩 onInboxMessages（消费方 await 内 callback / Runtime 不并发 batch）

**归本模块**：clawforum 常驻 agent 长期循环的唯一入口。业务模块要让 agent 「长期跑」必经本模块。

**不归本模块**：

- 模块装配（L1-L4 instances 构造），归 L6 Assembly
- agent 业务身份（motion / claw 分支），归 Assembly 注入 systemPromptBuilder + identityToolFilter（Runtime 内部不分支）
- 单步执行算法（一次 LLM 调用加 tool 派发），归 L3 StepExecutor
- agent 完整循环算法（多步调度加停止判定），归 L3 AgentExecutor（Runtime 是 caller / 写 stepCallback）
- 异步任务调度，归 L4 TaskSystem
- 契约状态机（生命周期管理加验收加重试），归 L4 ContractSystem
- 单步 LLM 调用容错，归 L2 LLMOrchestrator
- 工具实现加权限校验，归 L2 Tools
- dialog 持久化磁盘 I/O，归 L2 DialogStore（Runtime 是 caller）
- snapshot git 提交，归 L2 Snapshot

**不可消除耦合理由**：M#1 单一职责（事件驱动循环跟装配跟单步执行跟循环算法跟任务调度跟契约状态机都是独立可变职责）加 M#2 业务语义归属（事件循环加 turn 边界加中断响应加轮级 commit 加 dialog 落盘协调语义归本模块）derive 自 Design Principle「运行中断即从最后一次完整 LLM 调用恢复状态并继续」加 D11「motion 为 clawforum 中一个特殊的 claw」（Runtime 内部无身份分支 / Assembly 装配期注入 identity 差异 / KD#24）— Runtime 是 long-running service 的唯一发起点，把 agent 循环包装成可生命周期管理的常驻服务。

## Cron [capability, DI]

**生产方**：`l5_cron`

**消费方**：

- `l6_assembly`（DI，装配期注入 CronJob[] + 工厂构造 CronRunner）
- `l6_daemon`（间接 / Daemon spawn `node cron-entry.js` 独立进程 / SIGTERM 终止）

**接口签名**：

```ts
export class CronRunner {
  constructor(jobs: CronJob[], audit: AuditLog);
  start(tickIntervalMs?: number): void;   // default 1000ms
  stop(): void;
  tick(): void;                            // 单 tick 触发 / 测试 + 派生用 / 应然原 silent
}

export type CronSchedule =
  | { type: 'daily'; time: string }       // "HH:MM" 每天固定时刻
  | { type: 'hourly' }                    // 每小时整点
  | { type: 'interval'; minutes: number }; // 每 N 分钟

export interface CronJob {
  name: string;
  enabled: boolean;
  schedule: CronSchedule;
  handler: () => Promise<void>;
}

export function parseSchedule(s: string, audit?: AuditLog): CronSchedule;
// 格式：'hourly' | 'daily:HH:MM' | 'interval:Nm'
// 未知格式 → fallback hourly + audit `cron_parse_fallback` (audit 可选 / 应然原 silent on audit 参 / 实然加为 fallback observability)

// 工厂
export function createCronRunner(jobs: CronJob[], audit: AuditLog): CronRunner;
```

注：`AuditLog` 来自 [L2 AuditLog](./l2a.md#auditlog-capability-di)。**应然 silent on `tick()` 公共 method + `parseSchedule` 第二 audit 参**（实然 underspec / 应然 align）。

**使用语义**：

- `start()` 默认 1000ms tick / 可经入参覆盖
- `stop()` 调后 instance 不可再用
- CronJob[] 装配期注入 / 运行期不变（runner 不知具体业务 / 装配方注入 schedule + handler）
- runKey 去重靠 lastRunKey Map / 重启重置（runner 派生态不落盘）
- 同 job 防重叠靠 running Set / handler 进出 add/delete
- jobs 必须设计为幂等（重启场景下重跑无副作用）
- schedule 仅 3 形态 / 不解析复杂 cron 表达式
- 不预设跨 daemon 协调（per-runner instance）
- 失败语义：
  - parseSchedule 未知格式 → fallback hourly + audit `cron_parse_fallback`（不抛）
  - job handler 异常 → catch + audit `cron_job_error` / runner 不终止 / 继续下一 job
  - LLM 调用 / disk 阈值 / 业务通知等失败语义归各业务 job 自管（runner 不感知）

**归本模块**：clawforum 定时任务调度框架的唯一入口。业务模块要按时间表触发任务必经本模块。

**不归本模块**：

- 业务任务语义（具体 cron job 业务逻辑：dream-trigger / disk-monitor / llm-stats），归各业务模块自己 own / Cron 只 own 触发机制
  derive 自 M#5「底层模块不预设上层模块语义」
- 跨进程协调（per-daemon instance 不预设跨 daemon 协调）
- 持久化任务历史（Cron runtime ephemeral / 去重时窗状态重启 reset / D4 显式豁免）
- 任务结果回传（Cron 仅触发处理器 / 处理器内部业务归处理器 caller）
- 复杂 cron 表达式解析（仅 3 形态）
- LLM 调用主路径（仅 dream jobs 内部消费 / runner 不知）
- 异步任务派发（具体 cron job 内部若需派子代理走 L4 TaskSystem）
- 跨进程通信（disk-monitor 投 motion inbox 走 L2 Messaging InboxWriter）

**不可消除耦合理由**：M#1 单一职责（定时调度框架跟具体业务任务跟跨进程协调跟任务持久化都是独立可变职责）加 M#2 业务语义归属（tick 调度加 runKey 去重加 job 防重叠加 job 异常隔离语义归本模块）derive 自「让定时调度可靠」业务 invariant 加 M#5「底层模块不预设上层模块语义」— Cron 是 generic 调度 primitive，jobs 业务由装配方注入，让多种业务（记忆整合 / 监控告警 / 统计聚合）共享同一调度框架。

## Gateway [capability, DI]

> 数据面交互门面 / 与 CLI（控制面）并列。Gateway interface + class + factory / 5 method 实施（start / stop / askUser / getActiveConnections / isOnline）/ 详 [modules/l5_gateway.md](../modules/l5_gateway.md)。

**生产方**：`l5_gateway`

**消费方**：

- `l6_assembly`（DI / motion-only 装配）
- `l6_daemon`（间接 / interrupt 回调注入 / port pattern 同 LLMEventSink）

> 上游 deps（Gateway 调它们）：L1 Transport（可选 / 无则进入 offline mode）+ L2 Stream（streamFactory 注入）+ L2 AuditLog + L2 Tools（ask_user 工具注册）— 详见 arch 表 1 «依赖» 列。

**接口签名**：

```ts
export interface Gateway {
  start(): Promise<void>;                              // 绑 transport 回调 + 启 StreamReader / 重复调用抛错 / offline 下 no-op
  stop(): Promise<void>;                               // 先停 reader / drop 连接 / close transport / idempotent
  askUser(question: string, ctx: ExecContext): Promise<ToolResult>;  // Step 2 占位 (实然抛 Error 'ask_user not implemented')
  getActiveConnections(): readonly Connection[];       // 当前连接快照 / 不暴露引用
  isOnline(): boolean;                                  // online/offline 一次性定型
}

export interface GatewayInput {
  /** StreamReader 工厂；Gateway 注入 onEvent 回调后调用 start */
  streamFactory: (onEvent: (event: StreamEvent) => void) => StreamReader;
  /** 已处于 listening 状态的 Transport / undefined = offline */
  transport?: Transport;
  /** Daemon 注入的 interrupt 回调 */
  interrupt: (reason: 'user') => void;
  /** askUser 超时（Step 3 使用） */
  askUserTimeoutMs?: number;
  /** audit 写入器 / Gateway 用于写 10 类结构化事件 */
  audit: AuditLog;
}

// 工厂
export function createGateway(input: GatewayInput): Gateway;
```

详细 ClientMessage / ServerMessage discriminated union + ask_user 工具实装细节归 [modules/l5_gateway.md](../modules/l5_gateway.md) §1 + §3 + §5（实然 src/core/gateway/types.ts + gateway.ts + ask-user-tool.ts 4 文件实装 / Step 2 askUser 占位实施 / Step 3 完整实装待后续 phase）。

**使用语义**：

- online / offline 一次性定型（`isOnlineMode = (transport !== undefined)` 启动期定 / 运行期不变）
- offline 下 ask_user 立即返 failureResult
- **数据面 vs 控制面分离**：CLI 唯一对外入口语境限定**控制面**（daemon 生命周期）/ Gateway 承担**数据面**（运行中 agent ↔ 外部客户端实时交互）/ 两者并列不冲突
- interrupt debounce：窗口内重复 trigger drop（不 emit 不 drop 连接）
- 连接生命周期：onConnect 加入 connections Map / onDisconnect 移除 / dropConnection 主动清理（broadcast 失败 / malformed JSON / unknown type）
- 失败语义：
  - broadcast 失败 → audit `gateway_connection_dropped` + dropConnection
  - malformed JSON / unknown msg type → dropConnection + audit
  - ask_user timeout / abort → audit `gateway_ask_user_cancelled` + 返 failureResult

**归本模块**：clawforum 数据面交互门面（外部客户端 ↔ 内部系统实时交互）的唯一入口。

**不归本模块**：

- 消息业务语义（Transport 只传 opaque string / Gateway 仅 JSON parse + discriminated union dispatch），归各 caller
- 连接鉴权 / 限流，归 Transport 或未来安全层
- StreamEvent 格式，归 L2 Stream
- StreamReader 构造（通过 streamFactory 注入）
- backpressure buffer（Transport 层 best-effort / Gateway 不做高层重传）

**不可消除耦合理由**：M#1 单一职责（数据面交互门面跟 Transport 实现跟 Stream 实现跟控制面 CLI 都是独立可变职责）加 M#2 业务语义归属（askUser 加 stream 订阅 加 interrupt 路由 加 连接生命周期语义归本模块）derive 自 Design Principle「用户可以观察运行过程中的所有状态」加 D8「事件驱动」加 D11「motion 单向访问」— Gateway 是数据面唯一发起点，让外部客户端与系统实时双向交互，与 CLI 控制面并列不冲突。

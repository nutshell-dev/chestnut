## Daemon [capability, direct]

**生产方**：`l6_daemon`

**消费方**：

- `daemon-entry.ts` 进程 main shim（direct，唯一进程入口消费 `daemonCommand`）

> 上游 deps（Daemon 调它们）：L6 Assembly（assemble / disassemble）+ L5 Runtime 公共 API（processBatch / abort / retryLastTurn）+ L2 ProcessManager（lockfile 操作）+ L2 Snapshot（启动期 commit）+ L2 AuditLog — 详见 arch §28 表 1 «依赖» 列。

**接口签名**：

```ts
export function daemonCommand(name: string): Promise<void>;

export interface DaemonInboxConfig {
  pendingDir: string;
  fallbackTimeoutMs?: number;
}

export interface DaemonMotionExtensions {
  heartbeat?: Heartbeat;
  onInboxMessages?: (messages: InboxMessage[]) => Promise<void>;
}

export interface DaemonLoopOptions {
  // 核心驱动（5 必填平铺）
  runtime: ClawRuntime;
  agentDir: string;
  clawId: string;
  label: string;
  audit: Audit;
  // inbox 配置（必填子组）
  inbox: DaemonInboxConfig;
  // motion 扩展（可选子组 / claw 整体省略）
  motion?: DaemonMotionExtensions;
  // 流式 / 回调（2 平铺可选）
  streamWriter?: StreamWriter;
  onBatchComplete?: () => Promise<void>;
}

export function startDaemonLoop(options: DaemonLoopOptions): {
  promise: Promise<void>;
  stop: () => void;
};

export function waitForInbox(
  fs: FileSystem,
  audit: Audit,
  pendingDir: string,
  fallbackTimeoutMs: number,
): Promise<void>;
```

**使用语义**：

- 唯一进程入口由 `daemon-entry.ts` 调 `daemonCommand(name)` 触发
- `daemonCommand` 计算 `dir = clawforumDir/name` / 确保 status/ 子目录 / 写 pid（冲突上抛 `LockConflictError`）/ 调 assemble + startDaemonLoop / 注册 4 signal handler / 进程驻留
- `DaemonLoopOptions` 4 组结构：5 必填核心驱动（runtime / agentDir / clawId / label / audit）+ inbox 必填子组 + motion 可选子组（claw 整体省略）+ 2 平铺可选（streamWriter / onBatchComplete）
- shutdown idempotent（再调不抛 / disassemble 全序继续）
- driver / state 分离：Daemon 调 Runtime 仅 3 方法（processBatch / abort / retryLastTurn）/ 不消费 Runtime 内部状态
- 双层 handler（shim daemon-entry.ts + 内层 daemon.ts）双发 audit 区分极早期 vs 运行期崩溃
- 失败语义：
  - `LockConflictError` → preAssembleAudit `assemble_failed`（module=lockfile / phase=preconstruct）+ 友好提示 + `process.exit(1)`
  - 其他 assemble 失败 → preAssembleAudit `assemble_failed`（module=pre_assemble / phase=preconstruct）+ `process.exit(1)`
  - runtime.initialize 抛错 → auditWriter `assemble_failed`（module=runtime / phase=post_assemble_init）+ `process.exit(1)`
  - daemon-loop fatal → audit `daemon_loop_fatal` + `process.exit(1)`
  - LLM error 重试 → audit `daemon_loop_llm_retry`（指数 backoff / max LLM_MAX_RETRIES）+ 继续
  - interrupt poller 异常 → audit `daemon_loop_interrupt_poller_disabled` + 继续（poller 关闭）
  - uncaughtException / unhandledRejection（运行期）→ audit `daemon_crash` + `process.exit(1)`
  - uncaughtException / unhandledRejection（shim 极早期 / daemon.ts 未入）→ shimAudit `daemon_uncaught_exception` / `daemon_unhandled_rejection` + console.error + exit
  - SIGTERM / SIGINT → shutdown 闭包 → disassemble → unlink pid → `process.exit(0)`

**归本模块**：clawforum 进程生命周期管理的唯一入口。daemon 进程 main 加信号处理加 Assembly 装配触发加 Runtime 事件循环驱动加优雅关停加 lockfile 单实例保护必经本模块。

**不归本模块**：

- 模块装配，归 L6 Assembly（Daemon 调 Assembly 装好后获得 Instances，不知 module 内部细节）
- 事件循环内部 state，归 L5 Runtime（driver 在 Daemon / state 在 Runtime）
- 子代理派发，归 L4 TaskSystem
- 审计落盘机制，归 L2 AuditLog（Daemon 调 AuditLog 接口写）
- agent 业务身份语义，归 Assembly 装配期 inject（motion / claw）
- 业务命令路由，归 L6 CLI（Daemon 是被 daemon-entry 启动 / 不直接接 user input）
- 长期运行 service 的事件循环算法，归 L5 Runtime
- 单步执行加 agent 完整循环算法，归 L3 StepExecutor 加 AgentExecutor

**不可消除耦合理由**：M#1 单一职责（进程生命周期管理跟模块装配跟事件循环 state 跟子代理派发都是独立可变职责）加 M#2 业务语义归属（main 入口加信号处理加 lockfile 加优雅关停语义归本模块）derive 自 Design Principle「事后可审计」加「进程独占防多实例竞争」业务 invariant — Daemon 是物理层进程实现的唯一发起点，把 abstraction stack（L1-L5 装配好的 Instances）装成进程让事件循环可生命周期管理。

## CLI [capability, direct]

**生产方**：`l6_cli`

**消费方**：

- 用户终端（direct，shell 调 `clawforum <subcommand>`）
- 智能体（direct，经 exec 工具调 CLI 子命令 / motion 透过 exec 执行 CLI 命令管理 clawforum）

> 上游 deps（CLI 调它们）：L1-L5 各模块（每命令独立实例化所需子集）+ L2 AuditLog（cli_invoke / cli_failed audit）+ L6 Daemon（经 ProcessManager 操作进程）+ L4 ContractSystem（contract 子命令）+ L6 Watchdog（经 H9 WatchdogPort 调命令）— 详见 arch 表 1 «依赖» 列。

**接口签名**（实然 = commander program-style entry / 不立单一 cliMain/CliSubcommandHandler/CliArgs 抽象）：

```ts
// 主 entry: src/cli/index.ts
// commander 'program' API 模式 / 不导出单 cliMain(argv) / 模块顶层执行 program.parse(process.argv)

// 子命令分散在 src/cli/commands/*.ts / 每命令 named export / 装配期由 program.command(...).action(handler) wire

// 主要子命令 named exports (示例 / 完整清单见 src/cli/commands/)
export function initCommand(/* ... */): Promise<void>;                 // claw init
export function startCommand(/* ... */): Promise<void>;                // claw start
export function chatCommand(/* ... */): Promise<void>;                 // claw chat
export function stopCommand(/* ... */): Promise<void>;                 // claw stop
export function listCommand(/* ... */): Promise<void>;                 // claw list
export function statusCommand(/* ... */): Promise<void>;               // status
export function sendCommand(/* ... */): Promise<void>;                 // send

// motion 子命令 (src/cli/commands/motion.ts)
export function motionInitCommand(/* ... */): Promise<void>;
export function motionChatCommand(/* ... */): Promise<void>;
export function motionStopCommand(/* ... */): Promise<void>;

// contract 子命令 (src/cli/commands/contract.ts)
export function contractCreateCommand(/* ... */): Promise<void>;
export function contractCreateFromDirCommand(/* ... */): Promise<void>;
export function contractLogCommand(/* ... */): Promise<void>;
export function contractEventsCommand(/* ... */): Promise<void>;

// skill 子命令 (src/cli/commands/skill.ts)
export function skillInstallUserCommand(/* ... */): Promise<void>;
export function skillInstallClawCommand(/* ... */): Promise<void>;

// 错误类
export class CliError extends Error {
  readonly exitCode?: number;
}

export function handleCliError(err: unknown): never;  // top-level catch / 转 audit + process.exit
```

**应然 silent on `cliMain(argv)` 单函数 / `CliSubcommandHandler` type / `CliArgs` interface**（实然 = commander program 模式 / 子命令 named export 分散文件 / 应然原 3 抽象 type 应然幻象）。

**使用语义**：

- CLI 是外部对系统操作的唯一入口（user / motion 等都经 CLI / D9 derive）
- 命令进程短生命周期：每次 invoke 独立实例化所需 L1-L5 模块子集 / 调完即结束
- 不持久化 module instances / 不跑长期 agent loop
- 子命令族：daemon 生命周期（start / stop / status / init）/ contract 操作（create / pause / resume / cancel / log）/ claw / motion / watchdog 子命令 / chat-viewport TUI
- 子命令路由由 CLI 主入口分派 / 各子命令 own 自身参数解析加业务调用
- daemon 进程操作经 ProcessManager + lockfile 等接口（不 own daemon process state）
- motion 作为用户的代理透过 exec 执行 CLI 命令管理 clawforum / 不绕过 CLI 直接调内部模块
- status 工具是进程内调用 CLI 命令处理函数（不 spawn 子进程 / 见 architecture.md 表 3）
- CLI 操作事件经 AuditLog audit emit（`cli_invoke` / `cli_failed` 等）
- 失败语义：
  - 子命令失败 → audit `cli_failed`（command + reason）+ exit code 非零
  - daemon 不在跑（status / stop / send 等命令）→ 友好提示 + exit code 非零
  - `LockConflictError`（start 命令）→ 友好提示 + exit code 非零（不传染调用方）
  - 参数解析失败 → 打印 usage + exit code 非零
  - 子进程操作失败（exec / spawn）由各子命令 catch + 转 audit

**归本模块**：clawforum 外部操作的唯一入口。用户加智能体加 motion 操作系统加 daemon 生命周期管理加 contract 操作加 claw / motion / watchdog 子命令加 chat-viewport TUI 必经本模块。

**不归本模块**：

- 业务模块装配持久化，归各业务模块 createX 工厂加 L6 Assembly（CLI 是命令进程 / 调完即结束 / 不持久化 module instances）
- 长期运行循环，归 L6 Daemon（CLI 短生命周期）
- agent 业务流程，归 L4 TaskSystem 加 ContractSystem 加 EvolutionSystem 等
- daemon 进程内 state 管理，归 L5 Runtime 加 L6 Daemon（CLI 通过 ProcessManager 加 lockfile 等接口操作 daemon process）
- agent 业务循环，归 L5 Runtime（CLI 不跑 agent loop）
- 进程间通信协议，归 L1 Transport（CLI 经 Transport 抽象访问 daemon）
- watchdog 内部细节，归 L6 Watchdog（CLI 经 H9 WatchdogPort structural typing 消费）

**不可消除耦合理由**：M#1 单一职责（外部操作入口跟业务模块装配跟长期运行循环跟 agent 业务循环都是独立可变职责）加 M#2 业务语义归属（命令路由加参数解析加子进程交互语义归本模块）derive 自 Design Principle「外部对系统的操作通过 CLI 唯一入口」加「事后可审计」— CLI 是外部加 motion 操作系统的唯一受信入口，命令进程短生命周期保证不同 invoke 间无状态泄漏。

## Watchdog [capability, direct]

**生产方**：`l6_watchdog`

**消费方**：

- `watchdog-entry.ts` 进程 main shim（direct，进程入口调 `runWatchdogLoop`）
- `l6_cli`（CLI watchdog 子命令直 dep Watchdog 公共 export / 内部细节由 module visibility 控制）

> 上游 deps（Watchdog 调它们）：L1 FileSystem（watchdog-state.json / watchdog.pid / logs）+ L2 ProcessManager（claw alive 检测）+ L2 AuditLog + L2 Messaging.InboxWriter（drop crash_notification 到 motion inbox）+ L2 Stream（LLM_OUTPUT_EVENTS Set 判定 claw 不活跃）+ L4 ContractSystem（utils / collectContractEvents / 仅读 contract 创建时间窗）+ L6 CLI config — 详见 arch §30 表 1 «依赖» 列。

**接口签名**：

```ts
// 业务 API (位置 src/watchdog/watchdog.ts)
export function getWatchdogPid(): number | null;
export function isWatchdogAlive(): boolean;
export function getWatchdogEntryPath(): string;
export function runWatchdogLoop(): Promise<void>;    // 内部 entry 调
export function shutdownWatchdog(/* ... */): Promise<void>;
export function loadWatchdogState(): void;
export function saveWatchdogState(): void;
export function logWithAudit(/* ... */): void;

// CLI 命令 (位置分散 / 实然位置 src/cli/commands/ 而非 src/watchdog/)
export function startCommand(): Promise<void>;       // CLI `clawforum watchdog start`
export function stopCommand(): Promise<void>;        // CLI `clawforum watchdog stop`
export function statusCommand(): Promise<void>;      // CLI `clawforum watchdog status`
```

**CLI 直 dep Watchdog 公共 export**（同层 L6→L6 单向 / 内部细节由 module visibility 控制 / 历史 H9 WatchdogPort 抽象层 phase422 删 / 详 modules/l6_watchdog.md §M#6）。**应然 silent on `shutdownWatchdog`/`loadWatchdogState`/`saveWatchdogState`/`logWithAudit` 公共 export**（实然有 / 应然 underspec）。

**使用语义**：

- 独立进程（不装进 motion / claw daemon / 由 CLI `watchdog start` 派生独立运行）
- **非智能体**（无 LLM / 无推理 / 无 prompt）/ 是监督基础设施
- 观察：motion 存活轮询 / claw 崩溃检测（was alive → now dead 且持合约）/ claw 不活跃检测（有活跃契约但 LLM 事件 timeout）
- 干预：motion 自动重启（指数 backoff 上限 5 min）/ claw crash 通知中介（drop crash_notification 文件到 motion inbox）/ claw 不活跃提醒（drop 提醒到 motion inbox）
- 监控状态磁盘持久化（watchdog-state.json 跨重启共享 last seen / backoff / claw 状态）
- claw crash 通知必经 motion 中介（不直接通知 claw / motion 单向访问设计）
- watchdog.pid 自管（不归 ProcessManager 自注册体系 / 监督进程不应由被监督方管 PID）
- 主 loop 轮询模型（非事件驱动 / D8 N/A 显式豁免）
- CLI 直 dep Watchdog 公共 export 消费（L6→L6 同层单向 / phase422 后真合规 / 内部细节由 module visibility 控制）
- 失败语义：
  - 主 loop iteration 异常 → catch + audit `watchdog_cleanup_failed` / 隔离不冒泡 / 继续下一轮
  - 状态文件 load 失败 → catch + audit `watchdog_state_load_failed` / 静默回退默认状态
  - 状态文件 save 失败 → catch + audit on `watchdog_stop`（save_failed payload）/ 关停时不阻塞
  - crash notify drop 失败 → catch + audit `claw_crash_notify_dropped` / 跨 motion inbox 软失败
  - watchdog 进程 uncaughtException → shim 兜底 + audit `watchdog_crash`
  - SIGTERM / SIGINT → 关停 + audit `watchdog_stop` / 写 watchdog-state.json + unlink watchdog.pid

**归本模块**：clawforum 进程级健康监控的唯一入口。motion 存活轮询加 claw 崩溃检测加 claw 不活跃检测加 motion 自动重启加 claw crash 通知中介加 motion 不活跃提醒必经本模块。

**不归本模块**：

- LLM 调用，N/A（Watchdog 是非智能体本质）
- agent 业务流程，归 L3-L5 各模块（仅观察加干预 fs 状态加 lifecycle 事件）
- 装配 module 图，归 L6 Assembly（Watchdog 是独立进程入口 / 自己不需要装 module 集合）
- PID 自注册，归 L2 ProcessManager 自注册体系（watchdog.pid 自管 / 防 ProcessManager 加 Watchdog 互相监控的循环）
- 直接通知 claw，归 motion 中介（应然 rule / motion 单向访问设计）
- 契约状态机，归 L4 ContractSystem（仅读 contract 创建时间窗判定）
- 事件流持久化，归 L2 Stream（仅消费 LLM_OUTPUT_EVENTS const 判定 claw 不活跃）

**不可消除耦合理由**：M#1 单一职责（进程级健康监控跟 agent 业务流程跟模块装配跟 PID 自注册都是独立可变职责）加 M#3 资源唯一归属（watchdog.pid 加 watchdog-state.json 加 logs/watchdog.log 独占）derive 自 Design Principle「motion 为 clawforum 中一个特殊的 claw / 对其他 claw 有单向访问权」加「磁盘即权威 / 内存可派生但不能为权威」加「事后可审计」— Watchdog 跨 daemon 监控（不能装进 motion / claw daemon 内 / 监控自己有逻辑悖论），独立进程加非智能体本质保证监督基础设施可靠性。

## Assembly [capability, direct]

**生产方**：`l6_assembly`

**消费方**：

- `l6_daemon`（direct，Daemon 调 `assemble(config)` 取 `Instances` 加 `disassemble(instances, signal)` 反向清理）

> 上游 deps（Assembly 调它们）：L1-L5 各模块的 createX 工厂（Assembly 是装配汇聚点 / 必须知道所有被装配模块构造签名）+ L2 AuditLog（装配期事件 audit）— 详见 arch 表 1 «依赖» 列。

**接口签名**：

```ts
export type Identity = 'motion' | 'claw';

export interface AssembleConfig {
  identity: Identity;
  clawId: string;
  clawDir: string;
  globalConfig: GlobalConfig;     // 来自 clawforum.yaml
  clawConfig: ClawConfig | null;  // identity='claw' 时必填 / 'motion' 时为 null
}

export interface Instances {
  readonly runtime: ClawRuntime;       // identity 经 systemPromptBuilder + identityToolFilter 注入
  readonly streamWriter: StreamWriter;
  readonly snapshot: Snapshot;
  readonly processManager: ProcessManager;
  readonly auditWriter: AuditWriter;   // 注：实然 class 名 `AuditWriter` (alias of canonical `AuditLog` / 详 modules/l2_audit_log.md §A)
  readonly cronRunner?: CronRunner;   // motion only + config.cron.enabled
  readonly heartbeat?: Heartbeat;     // motion only + heartbeat_interval_ms > 0
}

// LockConflictError 物理迁 L2 ProcessManager
// 详 [interfaces/l2a.md](./l2a.md) ProcessManager 节 LockConflictError class export
// 本模块不再 own / Assembly assemble 抛 LockConflictError = 来自 PM acquireLock / Assembly 直接 re-throw 不 wrap

export function assemble(config: AssembleConfig): Promise<Instances>;
export function disassemble(instances: Instances, signal: string): Promise<void>;
```

**使用语义**：

- Assembly 会随模块数量增加而变大 / 但对外耦合界面恒定为「装配 + 反向清理」两动作 / 外部模块增减不影响调用方
- 装配期跨模块注入清单：
  - caller 类型 universe + 各 tool 允许哪些 caller 调用 → Tools
  - gitignore content（各模块 disk artifacts 知识）→ Snapshot
  - 跨模块回调（执行过程事件回调 / interrupt / Cron handler / TransportErrorEvent fan-out / LLM 业务事件审计等）
  - Skill multi-source dir paths（含 user dir + system dir）→ SkillSystem
- Snapshot 单实例约束：保证唯一 Snapshot 对象（同时出现在 `Instances.snapshot` + `RuntimeDependencies.snapshot` / 防 `recovery-snapshot` audit 重复 bug）
- audit 是观察通道 / 不是失败处理通道：assemble 不可预期失败必须抛给 Daemon 决策（process.exit）
- disassemble 失败不抛（关停过程已无消费者可决策）
- AuditWriter 不在 disassemble 内 close（TSV 追加写无 close 义务 / 保证 daemon_stop 写入磁盘）
- `Instances` readonly 值对象：Daemon 不得重新赋值字段（tsc 编译期保证）/ 调字段对象的方法允许
- L1 预制：`systemFs`（enforcePermissions=false）+ `clawFs`（enforcePermissions=true）/ 两者 baseDir=clawDir
- 跨模块回调注入时机：parentStreamLog + contractNotifyCallback 经 RuntimeDependencies 字段注入
- 失败语义：
  - `LockConflictError` → audit `assemble_lock_conflict`（clawId）+ 抛 `LockConflictError`
  - 某模块 constructor 抛错 → audit `assemble_failed`（module / phase / reason）+ 抛 `Error(cause)`
  - snapshot.init 失败 → audit `assemble_failed`（module=snapshot / phase=init）+ 抛
  - recovery-snapshot commit 失败 → audit `assemble_failed`（phase=recovery-commit）+ **不抛**（recovery 失败不阻塞启动 / 显式决策）
  - 跨模块回调注入失败 → audit `assemble_failed` + 抛
  - runtime.initialize 抛错 → audit `assemble_failed`（module=runtime / phase=post_assemble_init）+ Daemon `process.exit(1)`（兜底）
  - 所有步骤成功 → audit `daemon_started`（clawId / pid）
  - disassemble 某步抛错 → audit `disassemble_step_failed` + 继续下一步（全序继续）
  - disassemble 末尾 → audit `daemon_stop`（signal）

**归本模块**：clawforum 模块装配根的唯一入口。按 identity 分支决定装哪些模块加跨模块回调注入加 Instances 句柄集构造加反向拓扑关停加 Snapshot 单实例约束加 lockfile 冲突识别加 gitignore content 加 caller universe 加权限矩阵组装注入必经本模块。

**不归本模块**：

- 模块内部初始化（init() / loadAll() / archive() 等），归各模块自身业务语义（由 createX 工厂内部完成）
- Runtime 业务动作（session repair / resumeContractIfPaused 由 Daemon 调 / Assembly 仅装配 / 不参与 runtime）
- 错误回滚（构造途中失败抛错由 Daemon catch 加 process.exit 加 OS 回收资源 / 不调 disassemble 回滚）
- 长期运行 service，归 L6 Daemon（Assembly 是 init-time function / assemble 加 disassemble 调完即结束）
- agent 业务流程，归 L3-L5 各模块
- LockConflictError 类定义，归 L2 ProcessManager

**不可消除耦合理由**：M#1 单一职责（模块装配根跟模块内部初始化跟 Runtime 业务动作跟错误回滚跟长期运行 service 都是独立可变职责）加 M#3 资源唯一归属（Snapshot 单实例约束保证唯一 Snapshot 对象）加 M#5「底层模块不预设上层模块语义」（L6 → L1-L5 单向依赖 / readonly Instances 防 Daemon 反向修改字段引用）derive 自 Design Principle「事后可审计」— Assembly own 上层业务装配语义（caller universe / gitignore content / 跨模块回调注入），让底层模块（L1-L5）保持 generic 不预设上层 universe，对外耦合界面恒定为「装配 + 反向清理」两动作让外部模块增减不影响调用方。

- Daemon onInboxMessages 转调（Daemon 接 Runtime processBatch 内 onInboxMessages callback / 转调 ContractSystem.handleReviewRequest 等业务模块）
